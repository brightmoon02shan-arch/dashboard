# 休假关注名单（ABNORMAL_STATUS_REPORT）

> **⚠️ 重要：休假关注名单使用 `abnormal_status_report` 专用工具，禁止自行编写 SQL。**
> 工具内部完成圈定与人群分组，返回单段报告骨架。**无需填充分析解读**——本报告以名单展示为主，不需要管理者结论。
> **卡片粒度**：单 scope → 1 张卡片整段发送；**default scope → 拆为 2 张卡片（高潜 + 管理者 各一张）**，因飞书卡片单张最多渲染 5 个表格，default 下两组合计可达 6 个表格会触发渲染截断。
> **多 scope 触发**：当用户单条问题里同时涉及多个 scope（例如"高潜和高绩效的休假情况"）时，LLM 应**多次调用工具**，每次单一 scope，**逐张独立发送**，不得合并。

---

## 一、工具调用

```bash
mcporter call --raw-strings hr_talent_claw.abnormal_status_report \
  dept_name="手机部" scope="default"
```

| 参数 | 必填 | 说明 |
|------|------|------|
| `dept_name` | ✅ | 部门名称。**测试阶段仅支持 HW / AU / HC / IS / IT / HR / TC / FI 这 8 个一级部门及其任一层级子部门**（HW38 新业务部及其子部门除外） |
| `scope` | 可选 | 人群范围。`default`（默认，高潜+管理者互斥）/ `高潜` / `管理者` / `高绩效` / `应届` / `职级`。**每次调用只能传一个**；多人群叠加请多次调用并分别发卡 |
| `pos_lvl` | 可选 | 职级筛选，支持单值 `"18"` 或区间 `"17-19"` / `"17~19"`；给定时强制 scope=职级 |

### 🎯 场景识别（与 PRD §4.2 query 链路对齐）

| 用户问题（PRD query） | 工具参数 | 输出卡片数 |
|----------|---------|---|
| "XXX部门的特殊休假" | `dept_name="XXX"` | **2（高潜 + 管理者 各一张）** |
| "XXX部门的休假情况" | `dept_name="XXX"` | **2（高潜 + 管理者 各一张）** |
| "XXX部门高潜的休假情况" | `dept_name="XXX", scope="高潜"` | 1 |
| "XXX部门管理者的休假情况" | `dept_name="XXX", scope="管理者"` | 1 |
| "XXX部门高绩效的休假情况" | `dept_name="XXX", scope="高绩效"` | 1 |
| "XXX部门应届生的休假情况" | `dept_name="XXX", scope="应届"` | 1 |
| "XXX部门 17~19 级的休假情况" | `dept_name="XXX", pos_lvl="17-19"` | 1 |
| "XXX部门 19 级特殊休假" | `dept_name="XXX", pos_lvl="19"` | 1 |
| "XXX部门丧假/产假名单" | `dept_name="XXX"` | **2（高潜 + 管理者 各一张）** |
| "XXX部门需要关注的人" | `dept_name="XXX"` | **2（高潜 + 管理者 各一张）** |
| "XXX部门高潜和高绩效的休假情况" | 调用 2 次：`scope="高潜"` 一次、`scope="高绩效"` 一次 | 2（独立两张卡片）|

> **为什么 default 输出 2 张卡片**：飞书 markdown 卡片单张最多渲染 5 个表格；default 模式下高潜+管理者 各 3 个假种小节合计可达 6 个表格，会超限导致后段表格不渲染。LLM 必须按下面 §五 B 流程将工具返回的单段 content 拆为 2 张独立卡片发送。

> **职级范围识别**：用户问题中出现 "17-19 级"、"17~19 级"、"17到19级"、"17至19级" 等表述时，统一传入 `pos_lvl="17-19"`（半角连字符）。工具内部会校验并转为 SQL 区间查询。

### ⚠️ 自指性表述拒答（最高优先级）

当用户提问中未出现具体部门名称，而是使用"我部门"、"我们部门"、"本部门"等自指性表述时，**直接拒答**，禁止调用工具。

回复话术："休假关注名单不支持「我部门」等自指性表述，请提供具体的部门名称（当前仅支持 HW/AU/HC/IS/IT/HR/TC/FI 8 个一级部门及其子部门），例如「手机部特殊休假名单」。"

### ⚠️ 部门白名单（测试阶段）

**当前阶段只支持以下 8 个一级部门及其任一层级子部门**（HW38 新业务部及其子部门除外）：
- HW（手机部，**一级查询时**排除二级「新业务部」HW38；HW 树下任一层级均排除「生产制造序列」）
- AU（汽车部，AU 树下任一层级均排除「生产制造序列」）
- HC（大家电，HC 树下任一层级均排除「生产制造序列」）
- IS（互联网业务）
- IT（集团信息技术部）
- HR（人力资源部）
- TC（技术委员会）
- FI（财务部）

> **支持子部门查询**：例如 `IT-AI`、`HW01-销售运营部`、`HC02` 等任一层级子部门均可作为 `dept_name` 直接传入。工具会自动按部门解析层级（dept_id_lvl1~lvl6）匹配员工。

如果用户查询的部门一级祖先不在白名单内（例如查询「XX供应链」隶属的一级是 SC），工具会返回拒绝文案，LLM 直接原样输出，**禁止自行替换为白名单内的部门**。

### 不生成报告的情况

| 检查项 | 拒绝条件 | 工具返回的拒绝原因 |
|---|---|---|
| 部门未找到 | 部门名称无法解析 | 没有查到您「XXX」部门的权限，请确认部门名称是否正确。 |
| 部门歧义 | 部门名称匹配多个候选 | `{"error": "...", "candidates": [...]}` |
| 部门权限 | 调用者对该部门无可见权限 | 您（{sender_id}）没有查看「XXX」休假关注名单的权限，请联系管理员开通。 |
| 不在白名单 | 一级祖先不在 HW/AU/HC/IS/IT/HR/TC/FI 内 | 休假关注名单当前阶段仅支持以下一级部门及其子部门 ...，「XXX」暂不在范围内。 |
| HW38 显式拒绝 | 查询部门一级=HW 且二级=HW38（含子部门） | 休假关注名单当前阶段不支持「XXX」（HW38 新业务部及其子部门）。 |

LLM 收到拒绝文本后**直接原样输出**，不做额外修饰，不尝试编写 SQL 替代，不替换部门重试。

## 二、错误处理

如果工具返回 `{"error": ...}`（超时、查询失败等），**必须再次调用 `abnormal_status_report` 工具重试**（最多重试 1 次）。

**绝对禁止**：
- ❌ 工具返回错误后自行编写 SQL 查询
- ❌ 自行拼凑报告数据
- ❌ 调用 `sql_query` 替代

如果重试仍失败，向用户说明"休假关注名单工具暂时繁忙，请稍后再试"。

## 三、工具返回值

### 1. 成功

返回 JSON：
- `sections`：**1 段**（key=`report`）。含 `key`、`title`、`content`
- `full_report`：完整 Markdown 报告（与 `sections[0].content` 一致）

| key | title（工具返回） | content 结构 |
|---|---|---|
| `report` | 【XXX部】的休假关注名单 | 顶部：纯文本头部块（📅 统计周期 / 👥 员工类型 / 📊 样本覆盖，**不带 H 标题**，部门名仅在卡片 header 的 title 字段呈现）；中部：按人群分组（default 下含 `### 🎯 高潜` + `### 👔 管理者` 两小节互斥，单 scope 仅一节），组内按假种分小节（`#### 一、长期休假` → `#### 二、婚/丧假` → `#### 三、孕/产/育假`），表格列：姓名 / 基础信息 / 情况说明；尾部紧跟 `📊 口径说明`（婚/丧无门槛、长期 ≥10天、孕产育 ≥10天 三条）；最末（仅 default scope）："💡 还可查询：高绩效、近三年应届生、某职级的特殊休假情况，以及帮你分析具体某个人的近期整体情况。" |

> **default scope 的 content 必须由 LLM 拆为 2 张卡片发送**——见 §五 B。其余 scope 直接整段成卡。

**所有数据已由工具填充完毕，LLM 不需要做任何分析解读填充。**

### 2. 不满足生成条件

返回纯文本拒绝原因或带 `candidates` 的 JSON。LLM **直接输出**。

### 3. 错误

返回 JSON `{"error": "..."}`。按"二、错误处理"重试 1 次后告知用户繁忙。

## 四、LLM 输出规则

> **⚠️ 卡片粒度规则**
> - **单 scope**（高潜/管理者/高绩效/应届/职级）→ 1 张卡片，整段发送
> - **default scope** → **必须拆为 2 张卡片**（高潜 + 管理者 各一张），因飞书单张卡片最多 5 个表格，default 下两组合计可达 6 个表格会触发渲染截断
> - **多 scope 触发**（如"高潜和高绩效的休假情况"）→ 多次调用工具，每次单一 scope，各自成卡

### 红线规则

1. **禁止篡改工具返回的任何数据**（最高优先级）——表格中的姓名、OPRID、天数、假期类型、部门必须逐字原样输出
2. **禁止删减名单行**——表格里有多少人就输出多少人，禁止以"……"省略
3. **拆卡仅限 default scope**——按 §五 B 流程将 content 拆为"高潜"+"管理者"两张卡片；单 scope 不得再拆，多次调用结果也不得合并
4. **禁止补分析解读**——头部已含统计周期+样本覆盖，主体是名单，不要再加"建议关注"、"流失风险"等推测段落
5. **禁止暴露技术细节**——表名、字段名、SQL、`tag_dim_nm` 等技术信息不得出现在卡片正文中
6. **禁止编造工具未返回的指标**——不要自行计算"异常率"、"高潜请假占比"等比率
7. **禁止吞掉 default 末尾追问引导**——`💡 还可查询：…` 这一行必须原样保留在 default 第二张卡片（管理者卡片）的底部
8. **空名单原样输出"本期无命中。"**——不要补"暂无异常风险"等修饰；default 下即使某一组 0 命中仍发对应卡片

### 输出前自查

- [ ] 单 scope：卡片标题是否就是工具返回的 `sections[0].title`，无任何前后缀修饰？
- [ ] default scope：是否输出 **2 张** 卡片，标题分别是「【XXX部】高潜的休假关注名单」与「【XXX部】管理者的休假关注名单」？
- [ ] default scope：头部块（统计周期/员工类型/样本覆盖）是否在每张卡片正文顶部都保留了一份？
- [ ] default scope：💡 追问引导是否仅出现在第二张卡片（管理者）底部，第一张卡片不出现？
- [ ] 表格行数是否与工具返回完全一致（无截断、无省略）？
- [ ] 各假种小节标题（一、长期休假 / 二、婚/丧假 / 三、孕/产/育假）是否原样保留？
- [ ] 是否未自行添加分析解读 / 衍生指标 / 建议段落？

## 五、飞书卡片发送流程

### 飞书卡片发送规范（NON-NEGOTIABLE）

1. **工具选择**：飞书互动卡片必须使用 `feishu_im_message`（`action=send`、`msg_type=interactive`），**禁止**使用通用 `message` 工具发送 interactive 类型消息
2. **表格数量限制**：单张卡片内的表格总数（含 markdown 表格）有上限，超过会触发飞书 `ErrCode: 11310`，须拆成多张卡片
3. **拆卡规则**：每张卡片**控制在 3 个表格以内**，超过时按人群/分组拆分
4. **content 格式**：`feishu_im_message` 的 `content` 参数传**完整的卡片 JSON 字符串**（含 `schema`、`header`、`body` 三段），不是裸 markdown
5. **统一外观**：`template: "red"`、`schema: "2.0"`
6. **`receive_id`**：必须使用 runtime context 中的 `Chat ID`

> **本工具的拆卡映射**：默认每个 scope 最多 3 个假种小节（长期 / 婚丧 / 孕产育）即 ≤3 张表格，**单 scope 一张卡片够用**；default scope 含两个互斥人群组合计可达 6 张表格，**必须拆为 2 张卡片**（按下面 §B 流程，每张恰好 ≤3 张表格）。

### A. 单 scope（高潜/管理者/高绩效/应届/职级）— 1 张卡片

整段直接发送：

1. 读取 `sections[0].title`、`sections[0].content`
2. 组装完整卡片 JSON 字符串：
   ```json
   {
     "schema": "2.0",
     "header": {"title": {"tag": "plain_text", "content": "SECTION_TITLE"}, "template": "red"},
     "body": {"elements": [{"tag": "markdown", "content": "ORIGINAL_CONTENT"}]}
   }
   ```
3. 调用 `feishu_im_message`：
   ```
   feishu_im_message(
     action="send",
     msg_type="interactive",
     receive_id=<runtime Chat ID>,
     content=<上一步组装的卡片 JSON 字符串>
   )
   ```

### B. default scope — 拆为 2 张卡片

工具返回的 content 结构（按行段）：

```
📅 统计周期：...                      ← 头部块为纯文本（不带 H 标题，部门名仅在卡片 header 的 title 字段）
👥 员工类型：...
📊 样本覆盖：...

---

### 🎯 高潜                           ← 高潜段（含 1~3 个假种小节，每节 1 张表格）
#### 一、长期休假（N 人）
| ... |
#### 二、婚/丧假（N 人）
| ... |
#### 三、孕/产/育假（N 人）
| ... |

---

### 👔 管理者                         ← 管理者段（结构同上）
#### 一、长期休假（N 人）
| ... |
...

---

📊 口径说明
- 婚假/丧假：命中即纳入，无累计天数门槛
- 长期休假（带薪病假/长期病假/工伤假/年休假/事假）：窗口内累计 ≥10 天
- 孕/产/育假（产前检查假/产假/陪产假/哺乳假/育儿假/流产假）：窗口内累计 ≥10 天

---

💡 还可查询：高绩效、近三年应届生、某职级的休假关注名单，以及帮你分析具体某个人的近期整体情况。
```

**拆卡步骤**：

1. 以 `### 🎯 高潜` 为切点切割 content
   - **HEADER_BLOCK** = 切点之前的内容（即顶部 `📅/👥/📊` 三行纯文本头部块，到第一个 `---` 之前）
   - **HIPO_BLOCK** = 从 `### 🎯 高潜` 起、到下一个 `### 👔 管理者` 之前（不含分隔的 `---`）
   - **MGR_BLOCK** = 从 `### 👔 管理者` 起、到 content 结尾（含 `💡 还可查询...` 尾部）
2. 组装并发送 **卡片 1**（按 §A 第 2~3 步流程，使用 `feishu_im_message`）：
   - title：`【XXX部】高潜的休假关注名单`（XXX 取自工具返回的 dept_nm，与原 title 一致）
   - body markdown：`HEADER_BLOCK + "\n\n---\n\n" + HIPO_BLOCK`（≤3 张表格）
3. 组装并发送 **卡片 2**（同样走 `feishu_im_message`）：
   - title：`【XXX部】管理者的休假关注名单`
   - body markdown：`HEADER_BLOCK + "\n\n---\n\n" + MGR_BLOCK`（≤3 张表格；📊 口径说明 + 💡 追问引导自然包含在 MGR_BLOCK 末尾）
4. 两张卡片之间不输出额外文本，连续发送即可
5. **若仍收到 `ErrCode: 11310`**（极端场景：某组未来扩到 >3 个假种小节），按假种小节进一步拆卡，每张卡片仍 ≤3 张表格

> **空组处理**：若工具返回的 HIPO_BLOCK 或 MGR_BLOCK 表格全为空（仅有"### 一/二/三..."标题或显示"本期无命中"），仍按上述流程发送对应卡片，不得吞掉。

### C. 多 scope 触发

按 scope 逐个调用工具、逐张独立发送（每次都按 A 或 B 流程处理）：

```
loop scope in user_intent_scopes:
    result = abnormal_status_report(dept_name=X, scope=scope)
    if scope == "default":
        send_two_cards_per_section_B(result.sections[0])
    else:
        send_one_card_per_section_A(result.sections[0])
```

### 发送完毕

所有卡片发送完后，回复："已为您生成 {部门名称} 休假关注名单。"
- default scope：1 次工具调用 → 2 张卡片，回复改为"已为您生成 {部门名称} 休假关注名单（高潜、管理者各一张共 2 张卡片）。"
- 多 scope（M 次调用）+ 含 default：N = (default 次数)*2 + (其他 scope 次数)*1，回复"...共 N 张卡片"

**禁止**在文字消息中重复输出报告内容或 JSON。

## 六、数据维度覆盖（V1）

### 时间窗口
- 查询窗口：`[今天-31, 今天-1]`（昨天向前推 30 天）
- 命中条件：请假区间 `[start_tm, end_tm]` 与窗口有交集即命中（即 `start_tm ≤ T-1 AND end_tm ≥ T-31`）

### 假种分组与累计门槛

按下表三个组分别在窗口内累计天数（同员工跨假种在**同组内**累加），各组独立判定是否达标；分组的卡片归位仍取**优先级最高（数字最小）**的达标组（婚/丧 > 长期 > 孕产育）。

| 优先级 | 分组 | 命中假种 | 累计门槛 |
|---|---|---|---|
| 1 | 婚/丧假 | 婚假、丧假 | 命中即纳入（无门槛） |
| 2 | 长期休假 | 带薪病假、长期病假、工伤假、年休假、事假 | 累计 ≥10 天 |
| 3 | 孕/产/育假 | 产前检查假、产假、陪产假、哺乳假、育儿假、流产假 | 累计 ≥10 天 |

> **天数与假种展示**：员工"情况说明"列的**累计天数**为**所有达标分组**之和（不只是 top_cat），假种列表为所有达标分组的假种并集，按"婚假/丧假优先 → 分组优先级 → 字符顺序"排序。

### 人群定义

| 标签 | 来源 | 命中规则 |
|---|---|---|

| 应届 | `ads_ai_qa_emp_df` | `new_grdt_flg = 1` 且 `new_grdt_grd_cd` 在近 3 年（含今年）|

> **明细标签为单一人群标签 + 应届标识双值**：每行"基础信息"列第二行最多 2 段，分隔符顿号。
> - 第一段：当前 scope 命中的人群标签（高潜 / 管理者 / 高绩效 之一；应届/职级 通道下为空）
> - 第二段：应届标识（如 "23届应届生"），仅当员工同时满足近 3 年应届时输出
> - 示例："高潜、23届应届生" 或 "管理者" 或 "23届应届生"
> - **不会出现多人群标签拼接**（如 "高潜、管理者"），default scope 下高潜与管理者互斥分组各占独立小节。

### 计算口径
- **员工类型**：`real_emp_cls_cd IN ('101','115','118')`（正式 + 外包A + 外部顾问A），且 `emp_sts_cd = 'A'`（在职）
- **请假天数换算**：`lve_days + lve_hours / 8`；先在每个分组内累加判门槛，达标组并集后 `total_days` 取**所有达标组之和**
- **基础信息列**：末级部门（`dept_nm`）· 职位（`pos_nm`）· `pos_lvl` 级
- **scope=default 互斥规则**：先识别高潜，再从剩余人中识别管理者，避免重复
- **空值展示**：统一输出 `—`（全角破折号）
- **部门白名单（测试阶段）**：HW/AU/HC/IS/IT/HR/TC/FI 共 8 个一级部门；HW 排除二级「新业务部」；HW/AU/HC 排除「生产制造序列」
- **V1 范围**：仅识别特殊休假；预离职、出勤工时低位、活跃度下滑等其他异常类型预计在后续版本上线
