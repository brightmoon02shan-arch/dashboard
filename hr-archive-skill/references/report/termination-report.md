# 离职报告（TERMINATION_REPORT）

> **⚠️ 重要：离职报告使用 `termination_report` 专用工具，禁止自行编写 SQL。**
> 工具内部执行 8 个独立数据查询（关键指标整体 + 按月、预离职概况、月底人数 + 当月在职、绩优画像、离职原因、按部门、按职级），返回带数据表格的**报告骨架**。LLM 必须根据表格数据**填充所有结论和分析**，然后逐段发送飞书卡片。

---

## 一、工具调用

```bash
mcporter call --raw-strings hr_talent_claw.termination_report \
  dept_name="中国区" start_dt="2026-01-01" end_dt="2026-04-30" include_forecast=true
```

| 参数 | 必填 | 说明 |
|------|------|------|
| `dept_name` | ✅ | 部门名称，**支持任意层级（1~6 级，包括末级）**，如"中国区"、"手机部"、"深圳分公司"、"城市管理部"；也支持"小米集团"、"小米公司" |
| `start_dt` | ✅ | 本期开始日期，YYYY-MM-DD 格式（如"2026-01-01"）|
| `end_dt` | ✅ | 本期结束日期，YYYY-MM-DD 格式（如"2026-04-30"）|
| `include_forecast` | ✅ | 是否输出"二、人员预估"章节（bool）。规则见下方 [场景识别](#场景识别与参数映射) |
| `as_of` | 可选 | 向后兼容参数，未传 `start_dt/end_dt` 时按 YTD 模式（年初→as_of）处理 |

### 🎯 场景识别与参数映射

**LLM 必须根据用户 query 识别场景，传入对应参数。下表是 8 个制式场景：**

| # | 示例 query | 场景 | `start_dt` | `end_dt` | `include_forecast` |
|---|-----------|------|------------|----------|-------------------|
| 1 | "中国区离职报告" / "26年中国区离职报告"（未指定月份）| 当年 YTD | 当年-01-01 | 今天 | **true** |
| 2 | "26年中国区3月份离职报告" | 当年+明确月份（已过）| 2026-03-01 | 2026-03-31 | **false** |
| 3 | "26年中国区1-3月离职报告" | 当年+月段（已过）| 2026-01-01 | 2026-03-31 | **false** |
| 4 | "26年中国区3-6月离职报告"（假设当前 5 月，含未来月）| 当年+月段（含未来）| 2026-03-01 | 2026-06-30 | **true** |
| 5 | "25年中国区离职报告" | 过去年全年 | 2025-01-01 | 2025-12-31 | **false** |
| 6 | "25年中国区3月份离职报告" | 过去年+明确月份 | 2025-03-01 | 2025-03-31 | **false** |
| 7 | "25年中国区1-3月离职报告" | 过去年+月段 | 2025-01-01 | 2025-03-31 | **false** |
| 8 | 不在上述格式 | 非制式 | — | — | — |

**关键判断：`include_forecast` 怎么选**

- 若 `end_dt > 今天` 的日期（本期含未来月）→ `true`
- 若 `end_dt <= 今天` 的日期（本期全在过去）→ `false`
- 当年 YTD（未指定月份）视为含当前月 → `true`

**非制式场景（# 8）：禁止调用 `termination_report` 工具**。典型非制式包括：
- 跨年对比："24-25 年离职趋势"、"近三年离职率"
- Q 单位："上季度离职率"、"Q2 离职分析"
- 非标时间词："上半年"、"最近"、"年底前" 这类无法精确映射到日期
- 加额外筛选："中国区 13 级及以上的离职报告"（职级筛选工具不支持，可忽略职级按整体报告处理）

对非制式，LLM 应改用 `sql_query` 工具自由组织回答，不要强行套制式模板。

### ⚠️ 自指性表述拒答（最高优先级，必须在调用工具前判断）

**当用户提问中未出现具体部门名称，而是使用自指性表述时，必须直接拒答，禁止调用 `termination_report` 工具。**

自指性表述包括但不限于："我部门"、"我的部门"、"我所在的部门"、"我们部门"、"我们的部门"、"本部门"、"咱们部门"、"咱部门"等任何用"我/我们/本/咱"代替具体部门名称的表述。

**判断规则**：只要用户提问中的部门指代不是一个可识别的组织名称（如"中国区"、"手机部"、"互联网业务部"），而是代词性质的自指表述，一律视为未提供部门名称。

回复话术："离职报告不支持"我部门"等自指性表述，请提供具体的部门名称（支持任意层级，从一级到末级），例如"中国区离职报告"、"深圳分公司离职分析"、"城市管理部离职情况"。"

**禁止默认使用"中国区"或任何其他部门名称替代。禁止猜测用户所在部门。**

### 特殊场景处理

1. **用户未指定部门**：同上述自指性表述规则，**不调用工具**，引导用户提供具体部门名称。

2. **部门范围**：离职报告**不限制部门层级**，支持集团级、一级到六级（末级）所有部门。LLM 只需将用户提供的部门名称原样传给 `termination_report` 工具即可，不要自行猜测或过滤部门层级。

3. **"/"分隔的部门名称必须理解为层级关系**：当用户输入"A/B/C"格式（用"/"明确分隔层级）时，理解为"一级部门/二级部门/三级部门..."的层级关系。例如"人力资源部/中国区/COE"表示三级部门"COE"。将完整名称传给工具处理，由工具按层级路径解析。

4. **部门找不到时的处理**：
   - **部门名称未找到**：工具返回"没有查到您「XXX」部门的权限，请确认部门名称是否正确。"直接输出该拒答信息，**不做重试，不自行替换为其他部门名称**。
   - **部门名称歧义匹配多个**：工具返回 JSON 含 `candidates` 字段，LLM 应将候选列表展示给用户选择，禁止自动选择。

5. **职级条件忽略**：`termination_report` 工具不支持按职级筛选。若用户提问包含职级条件（如"中国区 13 级及以上的离职报告"），应忽略职级条件，直接调用 `termination_report(dept_name="中国区", ...)` 查询该事业部整体离职报告（工具自身会输出职级维度的下钻表）。

6. **时间场景识别**：见上方 [场景识别与参数映射](#场景识别与参数映射) 表，LLM 必须准确识别 query 的时间意图并传入对应的 `start_dt / end_dt / include_forecast`。不满足 8 个制式场景的，按"非制式"处理，改用 `sql_query`。

### ⚠️ 多轮对话隔离（必须遵守）

**每次离职报告请求必须独立处理，严禁混入对话历史中其他查询的数据或结论。**

- 填充分析解读时，**只使用当前工具返回的表格数据**，禁止引用对话历史中其他部门/其他查询的数据
- 如果用户连续查询了多个部门（如先查"中国区"再查"手机部"），每次报告的分析解读必须**完全独立**，仅基于当次工具返回的数据
- **同一查询重复提问必须结果一致**：对于相同部门、相同截止日的离职报告请求，无论是首次查询还是重复查询，工具返回的数据相同时，LLM 填充的分析解读结论也必须保持一致。禁止因上下文中已有的其他查询结果干扰当前报告的分析结论。如果对话上下文中已有该部门的报告，重新生成时必须**完全忽略之前的分析内容**，仅基于当次工具返回的数据独立填充

### ⚠️ 离职报告相关追问处理（必须遵守）

用户提问与离职报告相关、但并非直接要求生成报告时（如"XX 部门离职原因排名"、"绩优流失在哪些子部门"），按以下规则处理：

1. **前文已有该部门报告（上下文可见报告内容）**：可以基于已生成的报告内容作答。**数据必须与报告内容完全一致，严禁篡改任何数字或结论。**
2. **前文没有该部门报告**：**直接拒答**，回复："暂无该部门的离职报告数据，请先查询该部门的离职报告。"
3. **严禁先生成报告再利用报告内容回答问题**——如果用户没有主动要求生成报告，不得自行调用 `termination_report` 工具生成报告后再从中提取数据作答。

### ⚠️ 权限拒答后禁止自行替换部门（必须遵守）

- 如果工具返回"没有查看离职报告的权限"或"没有查看「XXX」部门离职报告的权限"，**直接输出该拒绝信息，立即停止**
- **禁止**在收到权限拒绝后自行替换为其他部门名称再次调用 `termination_report`
- **禁止**猜测或编造部门名称调用工具
- **禁止编造部门映射关系**：输出话术中不得出现"XX 是 YY 的默认映射部门"、"XX 默认映射为 YY"等编造的部门映射描述。系统中不存在任何部门间的"默认映射"机制，各部门名称独立使用，严禁自行臆造关联

### ⚠️ 多部门查询的权限处理（必须遵守）

**场景一：并列查询多个部门**（如"A 和 B 的离职报告"）

- 需要**逐个调用** `termination_report`
- **只要其中任何一个部门返回权限拒绝，立即停止所有后续调用**，直接输出权限拒绝信息，已成功生成的报告也**不发送**
- 拒绝信息应明确列出哪些部门没有权限（如"没有查看 B 部门离职报告的权限"），多个部门无权限则逐一列出
- 禁止"跳过无权限部门，只展示有权限部门"的做法——要么全部有权限才输出，否则整体拒答

**场景二：查询某部门下各子部门对比**（如"C 部门下各子部门的离职报告对比"）

- **父部门有权限后，逐一查询子部门**：对 C 下每个二级部门调用 `termination_report`，如果某个子部门返回权限拒绝，该子部门报告不生成，输出该子部门的权限拒绝信息
- 如用户想查看某一级部门下各二级部门的整体离职分布，可直接建议："C 离职报告 本身已包含各二级部门的离职率下钻表（4.1 按部门），可优先查看"

### 不生成报告的情况（拒答优先级最高）

工具会自动执行**前置校验**。当以下任一条件不满足时，工具返回拒绝原因（纯文本，非报告骨架），**LLM 应直接输出该拒绝原因，不要尝试自行编写 SQL 或生成报告**。

| 检查项 | 拒绝条件 | 工具返回的拒绝原因 |
|---|---|---|
| 全局权限 | 调用者无 HR 数据访问权限 | 您（{sender_id}）没有查看离职报告的权限，请联系管理员开通。 |
| 部门未找到 | 部门名称无法解析 | 没有查到您「XXX」部门的权限，请确认部门名称是否正确。 |
| 部门歧义 | 部门名称匹配多个候选 | `{"error": "'XXX' 匹配到多个部门...", "candidates": [...]}` |
| 部门权限 | 调用者对该部门无可见权限 | 您没有查看「XXX」的权限（或其它 resolver 返回的权限文本）|

**离职报告不限制部门层级**：一级~六级任意层级都可查询，无"部门层级拒绝"场景。

**LLM 处理规则：**
- LLM **直接原样输出**工具返回的拒绝文本，不做额外修饰、不尝试补充分析
- **禁止**在收到拒绝原因后自行调用 `sql_query` 或其他工具尝试绕过检查
- **禁止**在收到拒绝原因后替换部门名称重试

## 二、错误处理（必须遵守）

如果 `termination_report` 工具返回 `{"error": ...}`（包括超时、查询失败等任何错误），**必须再次调用 `termination_report` 工具重试**（最多重试 1 次）。

**绝对禁止**：
- ❌ 在工具返回错误后自行编写 SQL 查询
- ❌ 自行拼凑报告数据
- ❌ 调用 `sql_query` 替代

如果重试仍然失败，向用户说明"离职报告工具暂时繁忙，请稍后再试"，不得自行生成替代报告。

## 三、工具返回值

工具有三种返回情况：

### 1. 成功

返回 **JSON**，包含两个字段：
- `sections`：**4 或 5 个分段**（含 forecast 场景为 5 个：summary, core, forecast, active_analysis, breakdown；不含 forecast 场景为 4 个：summary, core, active_analysis, breakdown）。每段含 `key`（段落标识）、`title`（飞书卡片标题）、`content`（Markdown 内容）。sections 数量由工具根据 `include_forecast` 决定，LLM 严格按返回顺序逐张发送
- `full_report`：完整 Markdown 报告（所有 sections 拼接，用于飞书文档保存场景）
- 每个 section 的 `content` 中包含已填充数据的表格、核心结论（已有具体数字）、以及占位提示（括号包裹的说明文字，需 LLM 填充）

**工具负责数据查询、表格渲染和核心结论数据填充。分析解读由 LLM 根据表格数据生成。**

### 2. 不满足生成条件

返回纯文本拒绝原因（如 `没有查到您「XXX」部门的权限，请确认部门名称是否正确。`）或带 `candidates` 字段的 JSON（三级部门 / 歧义匹配场景）。LLM **直接输出**文本或展示候选列表，不做额外处理。

### 3. 错误

返回 JSON：
```json
{"error": "离职报告生成失败: ..."}
```

## 四、LLM 输出规则

> **⚠️⚠️⚠️ 最高优先级规则：离职报告必须以独立飞书卡片的形式逐张发送，严禁合并为一张卡片或一条文字消息！**
>
> 工具返回 JSON，`sections` 包含 **4 或 5 个段落**（视 `include_forecast` 场景而定）。LLM 必须对每个段落：填充占位提示 → 组装为独立的飞书卡片 JSON → 通过 `exec` 工具逐张发送。
> **每个 section 对应一张独立卡片，sections 有 N 段就发 N 次 `exec`，以工具返回为准。**
> 表格数据保持原样，结论和分析由 LLM 根据表格中的具体数字生成。

### 完整输出铁律（最高优先级，违反视为严重错误）

每个 section 的内容必须**完整、逐字输出**，禁止以任何理由截断、省略或缩减：

1. **表格必须完整输出所有行和列**——工具返回多少行就输出多少行，禁止只输出"前 N 行"或用"……"省略剩余行。即使表格很长（如按部门下钻表可能 30+ 行），也必须全部输出
2. **工具返回的所有 section 必须全部发送**（4 或 5 个）——禁止因响应长度压力而跳过任何 section；同时禁止自行补发工具没返回的 section（比如过去月段场景工具没返回 forecast，LLM 不得自己补一个 forecast 卡片）
3. **每个 section 内工具返回的所有表格必须全部输出**——`core` 默认 2 张表（1.1 整体 + 1.2 按月），**单月场景工具会仅返回 1.1**（按月表不适用），按工具实际返回的张数输出，不得缺漏；`forecast` 有 2 张表（预离职概况、月底人数），`active_analysis` 有 2 张表（绩优画像、离职原因），`breakdown` 有 2 张表（按部门、按职级），任何工具返回了的表都不得省略
4. **整体结论 + 各 section 分析解读必须全部填充**——禁止只输出表格不写分析，也禁止只写分析不输出表格
5. **禁止合并或简化表格**——不得将多张表合并为一张，不得删除"看起来重复"的列
6. **summary 卡片必须包含「统计口径」blockquote**——工具会在 summary content 末尾追加以下 blockquote 块（员工类型 + 绩优定义），LLM 组装 summary 卡片时必须**逐字原样保留**整个 blockquote（含标题行和两条明细行），禁止删除、禁止改写为纯文本、禁止剥离 `>` 前缀。即使工具因任何原因没返回该块，LLM 也应**按以下固定内容补齐**，确保每份离职报告的 summary 卡片都带统计口径说明：

   ```
   > **统计口径**
   > - 员工类型：正式+外包A+外部顾问A
   > - 绩优：最近一次绩效 B+ 及以上
   ```

7. **集团级（小米集团 / 小米公司）特有「数据说明」与「DEP 项目组注」必须原样保留**——当查询部门是集团级时，工具会额外注入两处 DEP 剔除说明：
   - **summary 卡片末尾**、「统计口径」blockquote 之后，追加一个独立 blockquote：
     ```
     > **数据说明**
     > DEP项目组（N人离职,离职率 X%）,属岗位结构性流动,剔除后集团实际离职率约 Y%（影响约 ±Zpp）
     ```
   - **breakdown 卡片 `### 4.1 按部门` 表格之后**、`### 4.2 按职级` 之前，追加一行 `>` blockquote：
     ```
     > 注：DEP项目组（N人离职,离职率 X%）,属岗位结构性流动,不反映业务部门人员稳定性。
     ```
   这两段由工具根据 `dept_field is None`（集团级）动态注入，**非集团级查询（任何具体部门）工具不会返回，LLM 也严禁自行补**。集团级查询到两段时，LLM 必须逐字原样保留（含 `>`、数字、标点、括号形式），禁止删除、改写、合并或剥离 `>` 前缀

如果担心输出过长，仍然必须完整输出——离职报告的完整性优先于长度控制。

### 🔴 红线规则（输出前必须逐条校验，违反视为严重错误）

以下红线适用于整篇报告所有「整体结论」和「分析解读」模块，**优先级最高**，任何分析解读在输出前必须逐条对照。

**红线 1：禁止篡改工具返回的任何数据（最高优先级）**

```
可识别去向的主动离职人员中，去向公司：字节（4人）、阿里（1人）、腾讯（1人）；去向行业：游戏、AI/大模型、创业各1人
```

规则：
- **仅用主动离职原因表格里已有的数据**，不允许调用其他 SQL 补数据
- 公司维度：按人数降序列出去向公司及人数
- 行业维度：列出去向行业方向
- **若表格里没有明确的去向公司/行业信息（即 `rsn_nm` 都是"个人发展"、"家庭因素"等原因类而非去向），直接跳过此条，不输出**
- 不要自己臆造公司名或行业

**分析解读规范**：
- `- ` bullet（**不带序号**），严格 3 条
- 每条必引用表格中的具体数字
- 第 1 条抓主要结论，第 2 条抓特殊/异常点，第 3 条抓建议/趋势
- **严禁编造工具未返回的衍生指标**（详见红线 8）。禁止出现"预离职率"、"绩优流失率"、"日均离职"、"环比 +X%"、"HC 缺口"等概念和对应数字。只能引用 8 张表里已有的指标

### 填充规范

- **整体结论和分析解读必须分条带序号**，不允许输出为连续大段文字
- **文中提及的日期必须与报告标题 / summary 中的 `start_dt ~ end_dt` 区间一致**，不得使用其他日期

- **禁止暴露技术细节**：表名、字段名、SQL、mock 等技术信息均不得出现
- **分析语言面向管理者**：用"稳定性下降"、"主动流失偏高"、"绩优流失集中"等业务语言

- **指标引用必须逐项核对，严禁张冠李戴**：在分析解读中引用数据时，必须仔细确认每个数字对应的指标名称、维度（今年/去年同期 / 同比）、口径（主动/被动/绩优/全员）和时间段完全一致。例如：不得把"去年同期离职率"的数字写到"今年离职率"的位置，不得把"绩优主动离职"的数字用在"主动离职总数"的描述中。**多部门对比场景尤其注意**：逐个部门填充分析时，必须核对当前引用的数字确实来自该部门的报告数据，严禁将 A 部门的指标数据错误引用到 B 部门的分析中

### 绝对禁止

- ❌ 将表格转为纯文字描述
- ❌ 省略工具实际返回的任何 section 或表格（若 `core` 返回 2 张表则必须输出 2 张；单月场景工具只返回 1.1，此时只输出 1.1 即为完整）

- ❌ 省略表格中的任何列
- ❌ 重新组织报告章节结构
- ❌ 删除或修改表格数据
- ❌ 自行编写 SQL 查询离职数据
- ❌ 输出中间推理过程
- ❌ 保留占位提示不替换

## 五、飞书卡片发送流程

> **⚠️ 离职报告必须使用 `msg_type=interactive` 发送飞书交互卡片，严禁使用 `msg_type=text` 或其他类型。**
>
> **⚠️ 所有卡片颜色必须一致**：一次离职报告的全部 4 或 5 张卡片的 `header.template` 字段**必须统一设置为 `"red"`**，不得随意换色，也不得给不同 section 用不同颜色。

### 发送步骤

对工具返回的 `sections` 数组（4 或 5 段，取决于 `include_forecast`），**按顺序逐个执行**以下操作：

1. **读取** section 的 `content`（Markdown 内容）
2. **填充**占位提示为实际分析文字（遵循上述红线和填充规范，**禁止篡改任何数据、禁止改变数值精度**——表格中是 17.4% 就写 17.4%，严禁缩减为 17%）
3. **组装**飞书卡片 JSON：
   ```json
   {
     "schema": "2.0",
     "header": {
       "title": {"tag": "plain_text", "content": "SECTION_TITLE"},
       "template": "red"
     },
     "body": {
       "elements": [
         {"tag": "markdown", "content": "FILLED_CONTENT"}
       ]
     }
   }
   ```
   - **颜色必须全部一致**：离职报告 4 或 5 张卡片的 `header.template` **必须统一为 `"red"`**，**禁止**给不同 section 用不同颜色（如 summary 用 red、core 用 blue），**禁止**换成 `purple`/`blue`/`green`/`orange` 等其他颜色——即便你之前看到 cost_report 用 `purple` 也不要混用。每次组装卡片时都写 `"template": "red"`
   - `SECTION_TITLE` = section 的 `title` 字段
   - `FILLED_CONTENT` = 已填充分析的 `content`（表格必须逐字原样保留，禁止修改任何数字、文字、小数位数、千分位格式）
   - **卡片标题必须包含【官方精选报告】前缀**：工具返回的 summary section 的 `title` 字段包含「【官方精选报告】」前缀，LLM 在组装飞书卡片时**必须原样保留**该前缀，严禁删除、修改或省略。发送前检查 summary 卡片的 `SECTION_TITLE` 是否包含「【官方精选报告】」，缺失则视为严重错误
   - **卡片标题不要在内容中重复**：`header.title` 已经是卡片标题，`FILLED_CONTENT` 中不要再以 `# 标题` 或加粗文字等形式重复一遍标题
   - **换行符必须保留**：`FILLED_CONTENT` 中的换行必须使用 `\n`（JSON 转义的换行符）。工具返回的 `content` 本身包含 `\n` 分隔各段落、表格行和分析条目，组装 JSON 时必须原样保留这些 `\n`，严禁将多行内容拼接成一行。如果换行丢失，飞书卡片将无法正确渲染 Markdown 格式（表格错乱、段落合并）
4. **使用 `exec` 工具发送卡片**（参考 `references/report/send-card.md` 的 `get_feishu_token()` + `send_card()` 函数）
5. `receive_id` **必须使用 runtime context 中的 `Chat ID`**（不是 `sender_open_id`）

### 发送顺序和卡片对应关系

工具返回的 `sections` 数组长度 **根据场景在 4 或 5 之间变化**。**章节中文编号（一/二/三/四）由工具动态分配**——forecast 缺失时后续章节会前移：

**场景 A：含 forecast（5 张）**
| # | key | 卡片标题 | 表格数 | 子章节号 |
|---|-----|---------|--------|---------|
| 1 | `summary` | 【官方精选报告】{部门}{年}年{月份}人员离职分析 — 整体结论 | 0 | — |
| 2 | `core` | **一**、关键指标 | 1 或 2 | 1.1（+ 1.2）|
| 3 | `forecast` | **二**、人员预估 | 2 | 2.1 / 2.2 |
| 4 | `active_analysis` | **三**、主动离职分析 | 2 | 3.1 / 3.2 |
| 5 | `breakdown` | **四**、维度下钻 | 2 | 4.1 / 4.2 |

**场景 B：无 forecast（4 张）**——章节编号整体前移
| # | key | 卡片标题 | 表格数 | 子章节号 |
|---|-----|---------|--------|---------|
| 1 | `summary` | 【官方精选报告】{部门}{年}年{月份}人员离职分析 — 整体结论 | 0 | — |
| 2 | `core` | **一**、关键指标 | 1 或 2 | 1.1（+ 1.2）|
| 3 | `active_analysis` | **二**、主动离职分析 | 2 | **2.1 / 2.2** |
| 4 | `breakdown` | **三**、维度下钻 | 2 | **3.1 / 3.2** |

> **`core` 表格数说明**：非单月场景（YTD / 整年 / 月段）工具返回 2 张表（1.1 整体 + 1.2 按月）；**单月场景**（如 "26 年 3 月份离职报告"）工具只返回 1.1，自动隐藏 1.2 按月表（单月拆不出按月趋势）。按工具实际返回张数输出即可。

**对应场景**：
- 场景 A（含 forecast）：当年 YTD、当年+含未来月
- 场景 B（无 forecast）：当年+过去单月/月段、过去年（全年/单月/月段）

**卡片标题中 `{部门}` 和 `{月份}` 部分由工具根据 query 参数动态生成**：

- `{部门}` = 工具解析后的部门全称（如 `中国区`、`手机部`、`小米集团-中国区-城市管理部`），**必须原样保留，不得省略**
- `{月份}` 由工具根据 `start_dt`/`end_dt` 计算：

| 场景 | query 示例 | `{月份}` | 完整标题示例 |
|------|------------|---------|-------------|
| 整年（含 YTD 截止 T-1）| "25 年中国区离职报告" / "26 年中国区离职报告" | 空串 | `【官方精选报告】中国区2026年人员离职分析 — 整体结论` |
| 单月 | "26 年中国区 3 月份离职报告" | `3月` | `【官方精选报告】中国区2026年3月人员离职分析 — 整体结论` |
| 月段（已过）| "26 年中国区 1-3 月离职报告" | `1-3月` | `【官方精选报告】中国区2026年1-3月人员离职分析 — 整体结论` |
| 月段（含未来月）| "26 年中国区 3-6 月离职报告" | `3-6月` | `【官方精选报告】中国区2026年3-6月人员离职分析 — 整体结论` |
| 过去年单月 | "25 年中国区 3 月份离职报告" | `3月` | `【官方精选报告】中国区2025年3月人员离职分析 — 整体结论` |
| 过去年月段 | "25 年中国区 1-3 月离职报告" | `1-3月` | `【官方精选报告】中国区2025年1-3月人员离职分析 — 整体结论` |

LLM 收到工具返回的 `title` 字段后**必须原样保留**，包括部门名、年份、月份段、`【官方精选报告】` 前缀、`— 整体结论` 后缀，**不得增删任何字符**。

> **⚠️ 卡片标题和子章节编号都由工具在 `content` 里生成，LLM 必须原样保留**——不得自己把"二、主动离职分析"改回"三、主动离职分析"，也不得改子章节号。按工具返回的 `sections` 顺序逐张发送。

### 发送完毕

所有卡片发送后，回复："已为您生成{部门名称}离职报告，共 N 张卡片。"（N = sections 实际长度，4 或 5）
**禁止**在文字消息中重复输出报告内容或 JSON。
**禁止**将多个 section 合并到一张卡片中——每个 section 必须是一张独立卡片。
**禁止**自作主张发 forecast 卡片（工具没返回就不发），也禁止跳过工具返回的任何卡片。

## 六、数据维度覆盖

| 分析维度 | 指标 | 覆盖 |
|---------|------|------|
| 关键指标 | 期末在职人数、已离职、离职率、主/被动、主动绩优、去年同期、同比 | ✅ |
| 预离职概况 | 按预计离职月分布（人数、主动、被动、绩优） | ✅ |
| 月底人数 | 当月在职（小标题下描述文字）+ 累计预计离职/入职/转出/转入 + 预估在职（当月底 / 次月底 / 次次月底 3 列）。**集团级场景**（小米集团 / 小米公司）下，累计预计转出、累计预计转入两行固定为 0（集团内部流转不算跨集团转岗），对应 SQL `pre_trsf_in` / `pre_trsf_out` CTE 的 WHERE 被替换为 `1 = 2`（永假）| ✅ |
| 绩优主动离职画像 | 按二级部门分布（绩优已离职、主动离职总数、绩优预离职、预离职主动总数） | ✅ |
| 主动离职原因 | 6 类原因（个人发展、继续学习深造、薪酬福利、城市选择、家庭因素、其他）+ 占比 | ✅ |
| 按部门下钻 | 按二级部门（在职、离职、离职率、主动、被动），按离职率降序 | ✅ |
| 按职级下钻 | 13-18 级（在职、离职、离职率、主动、被动） | ✅ |

### 计算口径说明

- **员工口径**（在职）：`real_emp_cls_cd IN ('101','115','116')` 且 `emp_sts_cd = 'A'`
- **流动口径**（离职）：`real_emp_cls_cd IN ('101','115','118')`
- **离职日期**：`tmn_eft_dt`（生效日期），禁用 `tmn_dt`
- **离职率** = `已离职 / (在职 + 已离职)`
- **绩优**：`emp_df.late_pfm IN ('B+','A','A+','S','S+')`（B+ 及以上，取最近一次绩效快照；避开 `pfm_f` 半/全年期期次结束日期差异）
- **主动离职** = `psv_tmn_flg = 0`；**被动离职** = `psv_tmn_flg = 1`
- **去年同期**：以 `end_dt` 的去年同月同日为截止（2/29 退位到 2/28），取 `start_dt ~ end_dt` vs `ly_start_dt ~ ly_end_dt` 的对比
- **空值展示**：数值列为 `NULL` / `0` / 分母为 `0` 时统一输出 `—`（全角破折号 em-dash，与 cost_report 一致）
- **SQL 1（1.1 关键指标整体）时间窗口特殊规则**（对齐 lizhi/0511需求.md L61-76）：**仅 SQL 1 的 start/end 与其他 7 条 SQL 不同**，其他 SQL 沿用通用参数不变。三种场景（以今天 2026-05-11 为例）：
  - **未指定月份（YTD）**，如"手机部 离职报告"：SQL 1 用 `2026-01-01 ~ 2026-05-10`（今天 T-1，避免今日数据不完整），去年同期 `2025-01-01 ~ 2025-05-10`
  - **往期月段 / 单月**，如"手机部 1-3月离职报告"：SQL 1 用 `2026-01-01 ~ 2026-03-31`（用户显式月末），去年同期 `2025-01-01 ~ 2025-03-31`
  - **往期整年**，如"手机部 2025年离职报告"：SQL 1 用 `2025-01-01 ~ 2025-12-31`，去年同期 `2024-01-01 ~ 2024-12-31`
  - 其他 SQL（按月、预离职、绩优画像、离职原因、按部门、按职级、月底人数等）：YTD 场景 end 仍按当月月末扩展，pre_cur_end_dt 扩到 T+2 月末，以覆盖预离职观察窗口
