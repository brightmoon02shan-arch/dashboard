# 入职洞察报告（ONBOARDING_REPORT）

> **🚨 两步流程，缺一不可：** ① 调用 `onboarding_report` → ② 填表写洞察 → 飞书文档 + **1 张**摘要卡片。  
> **调用工具后必须立即执行第 2 步**；禁止停下、仅输出文字摘要，或等待用户再指示。  
> **交付形态**：仅 1 张交互卡片 + 1 份飞书文档；禁令见 [§4.4](#44-红线规则输出前逐条校验)。  
> **🚨 卡片铁律（最高优先级）**：摘要卡片 **禁止独立撰写**；必须等 `FILLED_MARKDOWN` 写入飞书文档后，再从**同一份正文**摘录组装（见 [§4.3](#43-摘要卡片)）。

> **⚠️ 制式报告必须用 `onboarding_report` 专用工具**（禁止 SQL 拼凑）。工具返回 D1~D5 JSON；LLM 按 [report-field-mapping.md](onboarding/report-field-mapping.md) + [report-template.md](onboarding/report-template.md) 填充后交付。

## 一、工具调用

```bash
# Step 1: 调用 MCP 工具，结果落盘
FILE=$(/root/.openclaw/workspace/bin/mcp-call.sh mcporter call --raw-strings hr_talent_claw.onboarding_report \
  dept_name="中国区" start_dt="2026-01-01" end_dt="2026-04-30")

# Step 2: 校验 code===0 后拆分 JSON（禁止直接 read 完整工具返回，易触发截断）
PARTS_DIR=/tmp/report-parts
python3 /root/.openclaw/workspace/skills/hr-archive-skill/scripts/parse-report.py "$FILE" "$PARTS_DIR"
# 本地开发：python3 .claude/skills/hr-archive-skill/scripts/parse-report.py "$FILE" "$PARTS_DIR"

# Step 3: 按序逐个 read 分片
read("$PARTS_DIR/_meta.json")
read("$PARTS_DIR/d1_overall.json")
read("$PARTS_DIR/dep_impact.json")
read("$PARTS_DIR/d2_profile.json")
read("$PARTS_DIR/d3_probation.json")
read("$PARTS_DIR/d4_retention.json")
read("$PARTS_DIR/d5_attribution.json")
```

> **必做**：`parse-report.py` 拆为 7 个分片（含 `dep_impact`）→ 按上序 **逐个 read**（禁止一次 read 完整 `$FILE`）→ 内存合并为 `data` → 见 [§4.1](#41-端到端步骤)。

| 参数 | 必填 | 说明 |
|------|------|------|
| `dept_name` | ✅ | 部门名称，**支持 1~6 级（含末级）**；也支持「小米集团」「小米公司」 |
| `start_dt` | ✅ | 本期开始，YYYY-MM-DD |
| `end_dt` | ✅ | 本期结束，YYYY-MM-DD |

> `onboarding_report` **仅**接受上述三参数；**禁止**传入 `include_forecast` 等未定义参数。
>
> **部门名展示 SSOT（强约束）**：报告正文、飞书文档标题、摘要卡片 `header.title`、正文内 `{dept_name}` 相关占位符，**统一使用工具返回的 `data.dept_name`**（即 `{dept_display_name}`）。  
> **禁止**回退使用用户原始入参部门名。示例：入参 `dept_name="手机部"`，若返回 `data.dept_name="手机部(除新业务部)"`，所有对外展示必须写「手机部(除新业务部)」。

### 🎯 场景识别与参数映射

| # | 示例 query | 场景 | `start_dt` | `end_dt` |
|---|-----------|------|------------|----------|
| 1 | "中国区入职报告"（未指定时间）| 最近一年（滚动 12 个月）| 今天 − 1 年 | 今天 − 1 天 |
| 2 | "26年中国区3月份入职报告" | 当年+明确月份（已过）| 2026-03-01 | 2026-03-31 |

| 4 | "26年中国区3-6月入职报告"（含未来月）| 当年+月段（含未来）| 2026-03-01 | 2026-06-30 |
| 5 | "25年中国区入职报告" | 过去年全年 | 2025-01-01 | 2025-12-31 |
| 6 | "25年中国区3月份入职报告" | 过去年+明确月份 | 2025-03-01 | 2025-03-31 |
| 7 | "25年中国区1-3月入职报告" | 过去年+月段 | 2025-01-01 | 2025-03-31 |
| 8 | 不在上述格式 | 非制式 | — | — |

**#1 计算示例**：今天 2026-05-26 → `start_dt="2025-05-26"`，`end_dt="2026-05-25"`（`end_dt` 取昨天，避免当日数据不全）。

**#8 非制式**：禁止调用 `onboarding_report`，改用 `sql_query`。

### ⚠️ 自指性表述拒答（调用工具前）

「我部门」「我们部门」「本部门」等 → **禁止调用**。回复：「入职报告不支持「我部门」等自指性表述，请提供具体的部门名称（支持任意层级 1~6 级），例如「中国区入职报告」「深圳分公司入职分析」。」

### 特殊场景处理

1. **部门找不到**：直接输出工具拒答，不重试、不替换部门名。
2. **部门歧义**：展示 `candidates`，禁止自动选择。
3. **职级条件**：忽略职级，查部门整体。
4. **权限拒答**：直接输出并停止，禁止换部门重试。
5. **多部门并列**：逐个调用；任一权限拒绝则整体停止。

## 二、错误处理

| 类型 | 处置 |
|------|------|
| MCP 返回 `{"error": ...}`（非标准信封）| **重试 1 次**；仍失败 →「入职报告工具暂时繁忙，请稍后再试」 |
| `code !== 0` | **直接输出 `message`**；不读模板、不建文档、不发卡片；`4221` 展示候选部门 |
| 工具失败后 | **禁止** SQL 拼凑报告、禁止 `sql_query` 替代制式报告 |

## 四、填充与输出

### 4.1 端到端步骤

| 步骤 | 动作 | 输出物 | 本步禁止 |
|------|------|--------|----------|
| A | §一 拆分 read → 合并 `data` → 按 field-mapping + report-template **只写文档正文**；同时按 field-mapping §二 `{飞书文档标题}` 替换好所有占位符，产出 `DOC_TITLE` 字符串 | `FILLED_MARKDOWN` + `DOC_TITLE` | ❌ 禁止同时构思/撰写摘要卡片；❌ `DOC_TITLE` 禁止残留 `{}` 占位符 |
| A+ | **填表完整性核对**：逐分片（d1~d5）检查关键字段是否均已在 `FILLED_MARKDOWN` 中有对应值，缺一补一 | 补全后的 `FILLED_MARKDOWN` | ❌ 禁止跳过核对直接进入下一步 |
| B | [§4.4](#44-红线规则输出前逐条校验) 数据红线 + [§7.1](#71-信号选择标准选什么写) 洞察质量自查 | 定稿正文 | — |
| C-1 | [§五](#五飞书交付) ①~③ 创建文档并 **写入 `FILLED_MARKDOWN` 全文** | 文档 URL | ❌ 禁止在文档未写入前发卡片 |
| C-2 | **仅在此步**：从 `FILLED_MARKDOWN` 按 [§4.3](#43-摘要卡片) **摘录组装** `SUMMARY_CONTENT` → ④ 发卡片 | 1 张卡片 | ❌ 禁止重新分析、禁止改写措辞或数字 |
| D | 文字确认（仅此一处）| 「已为您生成{部门}新人入职报告：摘要卡片 1 张 + 完整文档 1 份。」 | — |

**禁止**在聊天消息中粘贴 `FILLED_MARKDOWN` 全文。

> **⚠️ 注意力提醒**：步骤 A 只负责把报告写进 `FILLED_MARKDOWN`；**不要**在 A 阶段「顺便」写卡片摘要。卡片是 C-2 的机械组装，不是第二次创作。文档与卡片各自独立撰写 = **流程错误**。

### 4.2 飞书文档约束

生成前 **必须先读** [report-template.md](onboarding/report-template.md)，严格按其中 **4 章 + 整体 callout（整体数据概览 / 分析解读 / 重点关注）** 结构填充。

| 约束 | 要求 |
|------|------|
| 篇幅 | 正文（不含表）2000~4000 字；第 1 章 ≤500 字 + ≤2 张表 |
| 分析 | 无异常维度不展开大段；每表下 `>` 结论须引用表中数字 |
| 章节 | D6/D7 本期不做；禁止跳过模板已有章节 |

callout、表格、占位符细则以 template 为准；数值填表以 field-mapping 为准。

### 4.3 摘要卡片

单张交互卡片（`msg_type=interactive`），`header.title` 与飞书文档标题 **必须与** [field-mapping §二](onboarding/report-field-mapping.md#二data-元信息) 一致（SSOT）。

#### 单一数据源（SSOT）

| 角色 | 内容 |
|------|------|
| **唯一正文源** | 步骤 A 产出的 `FILLED_MARKDOWN`（与步骤 C-1 写入飞书文档的全文**同一份**） |
| **卡片来源** | 从 `FILLED_MARKDOWN` **原文摘录** + 按 §4.3 demo 版式组装（允许加 `####` 标题、`---`、按句拆条、重排序号） |
| **禁止** | 为卡片单独写洞察、单独选信号、单独改措辞或数字 |

> **常见错误（必须避免）**：文档写完后凭记忆「概括」卡片 → 条数、措辞、数字与文档不一致。  
> **正确做法**：发卡片前**重新打开/对照 `FILLED_MARKDOWN`**，按下方映射表逐段复制，只做结构变换，不做内容变换。

#### 执行顺序（不可跳步、不可并行）

```text
A 写完 FILLED_MARKDOWN
  → B 红线自查
  → C-1 feishu_update_doc 写入全文
  → C-2 摘录 FILLED_MARKDOWN（1~5）+ 回读 dep_impact.json（6）→ 对照清单
  → ④ send_card
```

**C-2 组装步骤**（逐步执行，禁止跳步；版式 **必须** 对齐下方 demo）：

1. **整体块**：原样复制 `#### **整体数据概览**` 下 3 条 → 卡片同名标题下（**禁止**拆句或改字）
2. **整体分析**：原样复制 `#### 💡 **分析解读**` 整节（含标题与各条；卡片前插入 `---`）
3. **重点关注**：原样复制 `#### ⚠️ **重点关注**` 整节（2-3 条；卡片与文档标题一致）
4. **整体口径**：原样复制 callout 内 `> 数据口径` blockquote → 紧跟重点关注之后
5. **各章块**（第 1~4 章）：由文档 `## 第N章 {章名}` 生成卡片标题 `#### **第N章 · {章名}**`（**仅此一处允许改格式**）；其下原样复制该章 callout 内正文（从 `**▎数据概览**` 至 `> 数据口径` blockquote 结束）；章与章之间插入 `---`
6. **末尾**：发卡片前**回读** `dep_impact.json`；`dep_impact` 非 `null` 时输出 DEP「数据说明」（M/P/Q 见 [field-mapping §二-B](onboarding/report-field-mapping.md#二-bdep_impact字段有值时用于-im-卡片数据说明)，**不在** `FILLED_MARKDOWN`）+ `📄 [更多详细内容查看完整报告]({doc_url})`（**仅此两处允许新增固定文案**）
7. 执行「发卡片前对照清单」，全部通过后再 `send_card`

### 4.4 红线规则（输出前逐条校验）

**红线 0：计算必须反向验证**  
凡模型自行计算的差值/比率（如偏离值、倍数），必须做**反向验证**：算完后用结果反算回原值，验证等式成立才可写入。  
示例：`差值 = 45.0 - 21.7 = 23.3 → 验证：23.3 + 21.7 = 45.0 ✓`；`倍数 = 39 / 11 = 3.5 → 验证：3.5 × 11 = 38.5 ≈ 39 ✓`。验证不通过则重算。

**红线 1：禁止篡改工具返回数据（最高优先级）**  
表格数字、表头、章节号须原样输出。凡 `[]` 列表填表须与接口**顺序一致**；映射表「最多 N 行」时仅允许从数组头部顺序截取前 N 行。

        ↓

> **为什么需要 ①′**：不同 agent 版本绑定的飞书工具不同——低版本是 MCP 结构化 API（`title` 是显式字段，不易漏传），高版本是 CLI 桥接（`feishu_app_doc` 的 `args` 是扁平字符串数组，需要自己拼 `["docs","create","--title",DOC_TITLE]`，**漏传 `--title` 即 Untitled**）。仅靠指令文字无法消除该断裂，必须**事后回读 + rename 自愈**。

|------|------|

## 七、Voice 约束（全局洞察话术规范）

> ⚠️ **全局强约束（适用于所有 LLM 洞察输出——整体 callout 分析解读/重点关注、章节分析解读、子节结论）：**
>
> **通俗易懂，说明问题，不用情绪与主观评论。**
>
> - 通俗易懂：用业务方能直接理解的表达，不造词、不绕弯
> - 说明问题：每条结论必须指出"是什么问题"或"有什么影响"
> - 不用情绪：❌ "令人担忧""值得欣喜""形势严峻" → ✅ 直接说数据和后果
> - 不用主观评论：❌ "管理层应该反思""这说明重视不够" → ✅ 只说事实和归因
> - **禁止暴露内部结构**：洞察正文（含卡片）不得出现 `D1`~`D5`、`dev`、`|dev|`、`layer1`、`metric`、`归因维度`、`交叉归因` 等字段/分片名；冒号后只用**业务语言**写影响（见 §7.3）

### 7.1 信号选择标准（选什么写）

两节**分写、禁止合并**：

| 小节 | 条数 | 写什么 |
|------|------|--------|
| `#### 💡 **分析解读**`（**整体 callout**，摘要卡片同源） | **4-5 条** | 全报告 **TOP** 信号：发现了什么（数据 + 合法对比 + 影响） |
| `💡 **分析解读**`（**各章 callout**） | **2-3 条** | 仅本章维度；规则同 §7.1，不与整体 4-5 条重复角度 |
| `#### ⚠️ **重点关注**` | **2-3 条** | 须拍板/介入的风险：不管会怎样（**不与**上节重复） |

**整体 callout「分析解读」**（飞书文档顶部 + 摘要卡片 `#### 💡 **分析解读**`）按优先级从高到低选 **TOP 信号，写满 4-5 条**（禁止凑常识凑数；禁止重复「整体数据概览」数字）。仅当 P0~P3 合法信号穷尽且仍不足 4 条时，允许降至 3 条——**不得**为凑满 4 条降低 §7.1 质量红线。

**各章 callout「分析解读」**写 **2-3 条**，聚焦本章表格，标准同下，角度不与整体 TOP 4-5 重复。

| 优先级 | 类型 | 判断标准 |
|--------|------|----------|
| P0 | 反直觉 / 反常识 | 与 HR 日常认知相悖、且能用**合法对比基准**（见下）证实的数据 |
| P1 | 有行动指引 | 数据直接指向可核查的管理动作（须写明**具体分组 + 偏离方向**） |
| P2 | 跨章互证 / 矛盾 | 同一分组在 D2/D3/D4/D5 两处独立数据互相印证或冲突 |
| P3 | 相对突出（兜底） | 仅当 P0~P2 不足时选用：优先写 D5 `layer1`/`layer2` 中 **\|dev\| 最大** 且 `n≥3` 的分组；或 D1/D3/D4 **同期**组间差异最显著且能说明具体问题的一条。**禁止**用固定占比/倍数阈值（如 ≥40%、≥3x）判定是否入选 |

**禁止选择**：已知常识（如"外包流动率高""9月校招集中报到""校招定13级""双中心格局稳固"）、无法行动的背景信息、仅复述表格 Top1 占比且无偏离含义的句子。

#### 合法对比基准（SSOT；与 field-mapping 一致）

洞察里的「高/低/集中/异常」**必须**落在下列之一；**禁止**编造工具未返回的对比。

> **填表说明**：下表「数据来源」列仅供写稿时定位 JSON 路径，**禁止**原样写入飞书正文或卡片（读者看不到 D1~D5）。

| 基准类型 | 数据来源（禁止写入正文） | 正文写法要点 |
|----------|----------|----------|
| 对整体偏离 | D5 `dev` = 分组 `metric` − `overall.metric` | 「XX离职率45%，高于整体21个百分点」；优先 \|dev\| Top 分组 |
| 同期组间 | D1 `by_emp_type` 各类型 `tmn_rate_pct` 等 | 「外包B离职率71.7%，高于正式员工X个百分点」；可辅助用倍数**表述**，不作入选门槛 |

**明确禁止作为对比依据**：同比、环比、去年同期、上期均值、编制/HC/预算（入职工具无此字段）、模板示例数字（§4.4 红线 9）、无字段的「合同到期」「积压」「新人体验」。

**洞察质量三条红线**：

1. **❌ 禁止把业务规则当发现**：制度/流程/季节本身决定的现象不算洞察。
   - ❌ "校招全部集中在13级"——13级本就是校招定级
   - ❌ "9月入职占59%：校招批量报到"——季节形态是常识，不是发现
   - ✅ 须写偏离或互证：「社招转正均分低于整体2.1分，社招侧入职质量偏弱」

2. **❌ 禁止无数据/无业务常识支撑的因果推断**：归因须有数据字段或**强业务常识**支撑（外包合同周期、校招定级等行业级既定规律视为强常识）。
   - ❌ "待安排答辩58人：答辩积压影响新人体验"（无字段、无强业务常识）
   - ❌ **状态分布中的中间态**（待安排答辩、进行中、待审核）不能反推流程节奏：「23.1%待安排答辩、71.1%已通过：排期积压、试用期筛选节奏偏后置」——「待安排」混合了**未到转正节点**与**已到但未排**，仅凭比例无法区分；要写"积压""节奏后置"须有**入职日期—答辩日期时长字段**或**转正前后离职对比**支撑（无该字段时直接禁写）
   - ✅ 强业务常识支撑（外包合同短期、批量到期为已知用工特征）：「外包B离职566人、离职率63%：合同到期批量调整，不是管理问题」
   - ✅ 无原因字段：「外包B离职率71.7%，高于正式员工X个百分点；离职原因是否集中到期须对照原因表验证」
   - ✅ 有 `tmn_reason_top12` 支撑：「离职 Top1 为合同到期（占X%）」
   - ✅ 有转正前后对比支撑「节奏后置」：「转正后被动离职41人、为转正前12人的3.4倍：试用期筛选偏后置」

3. **❌ 禁止把"大"当"异常"**：占比高/人数多不等于有问题。
   - ❌ "零售研发部123人居首（23.6%）"——最大部门人数多属正常
   - ❌ 用「≥40%」「≥3x」等**未由业务定义的阈值**判定入选
   - ✅ 「二级部门A离职率45%，高于整体21个百分点」——有**当期**对比基准

**重点关注**（2-3 条）筛选：须同时满足——不干预下季度会恶化 × 影响一类人以上 × 会反复发生；与「分析解读」角度错开（本节只写后果，不重述发现）。

### 7.2 简洁原则

1. **说完即止**：一条结论只说一个观点，说清楚就停，不展开不重复不铺垫
2. **大白话优先**：用读者不需要二次翻译的表达。禁止自造隐喻（如"脉冲式入职""结构性数据扰动""人员流动的主战场"）
3. **数字克制**：每条只保留最关键的 1-2 个数字佐证，不堆砌
4. **一因即收**：给出原因或影响后立即结束，不做层层推演

### 7.3 句式公式

```
N. **[直白结论]**：[关键数字] + [一句话说明原因或影响]。
```

> 注意：序号 `N.` 不加粗，结论部分用 `**...**` 加粗，冒号后的数据和原因不加粗。

要求：
- **结论**：用最简单的主谓宾说清楚"什么情况"，禁止修辞包装
- **数字**：只保留最关键的 1-2 个数字，不堆砌
- **原因/影响**：一句话收尾，说明"为什么"或"意味着什么"；**只用业务用语**（部门名、职级、招聘类型、员工类型、转正前后等）

**冒号后禁止出现**（违反视为格式错误）：

| 禁止写入正文 | 应改为 |
|--------------|--------|
| `D1`~`D5`、`dep_impact` | 不写；用「员工类型」「第4章归因表」等业务指代或不标注来源 |
| `dev`、`\|dev\|`、`layer1`、`metric`、`residual`、`drilldown` | 不写；用「高于/低于整体X个百分点」「偏离最大」 |
| `D5职级维度`、`D5交叉归因`、`D1同期组间对比` | 不写；直接写「18级社招」「某二级部门」「外包B vs 正式」 |
| `归因维度`、`单因子`、`交叉归因` | 不写；写具体分组名 + 指标 |
| `IC`、`Individual Contributor`、`个人贡献者`、`高级别IC` | 不写；统一用「非管理者」（非HR读者无需翻译）。同类禁用：HRBP术语、英文缩写堆砌、其他场景（如 ES 搜索字段）渗透过来的内部行话 |

### 7.4 Few-shot（输出必须对齐此风格）

> 以下为**整体 callout「分析解读」**合规范本（**4-5 条**，下列示 5 条）——数字须来自当期工具返回；**冒号后不得含 D1~D5/dev**，填表时替换为真实分组名与数值：

```
1. **外包B离职566人、离职率63%**：合同到期批量调整，不是管理问题。
2. **18级社招离职率29.9%，高于整体8.2个百分点**：该职级社招是离职偏高的主力人群，入职适配风险集中。
3. **社招离职率25.1%，高于校招11.5个百分点**：社招留存压力明显大于校招。
4. **高薪offer离职率31.5%，高于普通offer 11个百分点**：高薪引入人才的留存效果不及预期。
5. **转正后被动离职41人，为转正前12人的3.4倍**：转正后淘汰多于转正前，试用期筛选偏后置。
```

> **第4章 callout「分析解读」**（2-3 条，同样禁止 D1~D5/dev）示例：

```
1. **中国区销服数字化部离职率40%，高于整体18.4个百分点**：该部门30人中12人离职，留存风险突出。
2. **19级非管理者离职率85.7%，高于预测58.2个百分点**：7人中6人离职，该群体留存几乎失效。
3. **国际销服数字化部转正均分3.14，低于整体0.31分**：该部门新人转正质量偏低。
```

> **对照——禁止入选的写法**（触犯 §7.1 红线或常识，**不得**模仿）：
> - ❌ "9月集中入职844人占全年59%：校招批量报到，非持续招聘增长。"
> - ❌ "校招全部集中在13级：只补基层，未补中高层骨干。"
> - ❌ "待安排答辩58人：答辩积压影响新人体验。"
> - ❌ "23.1%正式新人答辩待安排（80人），仅71.1%已通过：答辩排期积压明显，试用期筛选节奏偏后置。"（中间态状态分布不能反推积压或节奏后置；「节奏后置」专属于「转正前后离职对比」场景，不可挪用）
> - ❌ "外包B离职率71.7%：**D1同期组间对比**，流动率远超…"（暴露内部分片名）
> - ❌ "18级社招离职率29.9%：**D5职级维度|dev|最大组**…"（暴露 dev/维度标签）
> - ❌ "19级非管理者离职率75%，4人中3人离职：**高级别IC**留存几乎失效…"（"IC/Individual Contributor/个人贡献者"是HR内部行话，非HR读者需二次翻译；从 ES 搜索字段说明渗透过来——报告统一用"非管理者"）

> **对照——禁止的啰嗦包装**（即使数据正确也要避免）：
> - ❌ "事件性入职——9月单月844人占全年59%：…结构性数据扰动，业务侧需先做'事件解释'再做趋势判断。"

### 7.5 原因/影响要求

| 层次 | 说明 | 示例 |
|------|------|------|
| ❌ 禁止 | 贴标签 / 重述数据 / 常识解释 | "属典型的校招集中入职" / "外包流动率高" / "筛选机制较为严格" |
| ❌ 禁止 | 堆砌修辞 | "结构性数据扰动，业务侧需先做'事件解释'再做趋势判断" |
| ✅ 要求 | 用大白话说**当期对比** | "高于整体21个百分点" / "转正后淘汰为转正前3.5倍" |
| ✅ 要求 | 用大白话说影响 | "试用期筛选偏后置" / "答辩排期压力集中在X部门" |

**一句话说清原因或影响即可，不要包装。**

### 7.6 句式多样化

- 禁止超过 2 条用同一句式结构
- 变换责任主体 / 动作 / 对象组合，避免模式化输出
- 至少 1 条含前瞻判断（"若…则…"/"未来…"）

### 7.7 数据表达规范

- **百分点**写全称"个百分点"，不用"pp"缩写
   - ❌ "高出 16.5pp" → ✅ "高出16.5个百分点"
- **偏离值**用"高于/低于整体N个百分点"表达，或直接写两个数字让读者自己比
   - ❌ "偏离+4.3" → ✅ "高于整体4.3个百分点"或"该组25.4%，整体21.1%"
- **能用倍数说清的优先用倍数**（仅作表述方式，**不是**洞察入选阈值）
   - ❌ "高出18.9个百分点" → ✅ "高出近一倍"（当两组率差异直观时）
   - 禁止为凑「3 倍」而强行写倍数对比

### 7.8 Voice 负面约束（铁律）

1. **❌ 序号格式**：序号本身不加粗，结论部分必须加粗。正确格式：`1. **结论**：数据+原因。` 禁止写成 `**1. 结论：**数据` 或全不加粗
2. **❌ 禁用术语**：钩子、量级 outlier、形状反常、4 步法、参照系、漏斗、抓手、拉齐、赋能、打法、pipeline、解释力最强、解释力显著、**D1~D5**、**dev**、**|dev|**、layer1/2/3、归因维度、交叉归因（作正文标签时）
3. **❌ 禁止自造隐喻**：事件性入职、脉冲式招聘、结构性数据扰动、经验嫁接、人员流动主战场、压倒性主导——这些都是废话包装
4. **❌ 禁止空洞归因**："XX维度解释力最强""XX维度影响最显著"——必须说出具体是哪个分组、偏高还是偏低、说明什么具体问题
5. **❌ 禁止委婉**："一定程度上"、"或许值得"、"可能需要"——有判断就直说
6. **❌ 禁止空喊**："建议关注"、"值得重视"、"后续跟进"——必须说关注什么、为什么
7. **❌ 禁止堆砌数字**：一条结论里最多 2 个数字，多了读者记不住
8. **❌ 禁止情绪化表达**："令人担忧""值得欣喜""形势严峻""不容乐观"——直接说数据和后果
9. **❌ 禁止主观评论**："管理层应该反思""重视程度不够""需要高度警惕"——只说事实、数据、归因
10. **❌ 禁止长句**：说完即止，不展开不重复
11. **❌ 禁止把制度规则当洞察**："校招定13级""试用期6个月"等业务规则本身不是发现——要说就说规则之外的异常
12. **❌ 禁止无据归因当事实**：数据没显示原因时，用"可能""推测"标注，不直接断言；有明确字段支撑时才可陈述因果
13. **❌ 禁止把"占比大"直接等于"有问题"**：大部门人多、校招季入职多都是正常的，只有**合法对比基准**下的偏离才值得说
14. **❌ 禁止编造对比维度**：同比、环比、编制/HC、上期均值、≥40%/≥3x 等固定阈值——工具与业务均未提供则不得写

## 八、动态维度表格规则（§4.1.2 / §4.2.2 共用）

> 以下规则适用于第 4 章中所有按 `layer1` 动态展开的单维度表格。

1. **维度名来源**：小标题与表头首列均用接口返回的 `layer1` 的 key，但需做以下显示转换：
   - `dept_nm_lvl1` → `一级部门`
   - `dept_nm_lvl2` → `二级部门`
   - `dept_nm_lvl3` → `三级部门`
   - `dept_nm_lvl4` → `四级部门`
   - `dept_nm_lvl5` → `五级部门`
   - `dept_nm_lvl6` → `六级部门`
   - 其他 key 直接使用原值（如 `职级`、`招聘类型`、`是否管理者`）
   - **禁止将 `dept_nm_lvl*` 原样作为标题或表头输出**
2. **行首列**：填该维度下的 `group` 取值（如 `17`、`社招`、`是`）
3. **结论行不可省略（铁律）**：每个维度的表格上方**必须**输出一句结论，格式为 `- [结论内容]`，指出哪个分组偏离最大及归因方向。**禁止连续输出两张表格之间没有结论行。**
4. **逐维度自检**：输出完全部维度后回查——每个 `- **{维度名}**` 下方是否紧跟 `- [结论]` 行，缺任何一个维度的结论行则属于严重格式错误，必须补回。
5. **全量展开**：须按接口 `layer1` 全部 key 逐维建表，禁止合并维度、省略维度或把首列统一写成「分组」
6. **空数据处理**：
   - `layer1` 为空对象 `{}` → 写「单维度对比：暂无满足数据要求的维度。」，跳过全部表格
   - 某维度下为 `[]` → 该维小标题下写「该维度暂无足够数据」，不建表
7. **离职率归因专用**：离职率归因不含「离职类型」维度
