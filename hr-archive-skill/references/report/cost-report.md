# 成本报告（COST_REPORT）

> **🚨🚨🚨 成本报告是两步流程，缺一不可，禁止在任何一步停下：**
> 1. **调用 `cost_report` 工具** → 获取报告骨架（sections JSON）
> 2. **填充分析 + 发送 7 张飞书卡片** → 对每个 section 填充占位提示，然后逐张发送飞书卡片
>
> **调用工具后必须立即执行第 2 步。** 禁止在调用 `cost_report` 工具拿到返回值后就停下、等待用户指令、或仅输出一段文字摘要。工具返回的是半成品骨架，不是最终报告——LLM 必须填充分析并发送卡片，这才是完整的成本报告流程。

> **⚠️ 重要：成本报告使用 `cost_report` 专用工具，禁止自行编写 SQL。**
> 工具内部执行 12 个独立数据查询，返回带数据表格的**报告骨架**。LLM 必须根据表格数据**填充所有结论和分析**，然后逐段发送飞书卡片。

---

## 一、工具调用

```bash
mcporter call --raw-strings hr_talent_claw.cost_report dept_name="中国区" month="202603"
```

| 参数 | 必填 | 说明 |
|------|------|------|
| `dept_name` | ✅ | 部门名称，仅支持**一级部门或二级部门**（如"中国区"、"手机部"、"互联网业务部"），也支持"小米集团"、"小米公司"查询集团整体 |
| `month` | 可选 | 报告月份，YYYYMM 格式（如"202603"），默认当月 |

### ⚠️ 自指性表述拒答（最高优先级，必须在调用工具前判断）

**当用户提问中未出现具体部门名称，而是使用自指性表述时，必须直接拒答，禁止调用 `cost_report` 工具。**

自指性表述包括但不限于："我部门"、"我的部门"、"我所在的部门"、"我们部门"、"我们的部门"、"本部门"、"咱们部门"、"咱部门"等任何用"我/我们/本/咱"代替具体部门名称的表述。

**判断规则**：只要用户提问中的部门指代不是一个可识别的组织名称（如"中国区"、"手机部"、"互联网业务部"），而是代词性质的自指表述，一律视为未提供部门名称。

回复话术："成本报告不支持"我部门"等自指性表述，请提供具体的部门名称（支持一级部门或二级部门），例如"中国区成本报告"、"手机部3月成本报告"。"

**禁止默认使用"中国区"或任何其他部门名称替代。禁止猜测用户所在部门。**

### 特殊场景处理

1. **用户未指定部门**：同上述自指性表述规则，**不调用工具**，引导用户提供具体部门名称。

2. **部门范围限制**：成本报告仅支持集团级、一级部门、二级部门。**但 LLM 禁止自行判断部门层级**——必须将用户提供的部门名称原样传给 `cost_report` 工具，由工具内部判断层级并返回对应的提示信息。LLM 不得根据部门名称的"看起来像几级"自行拒答。例如"集团信息技术部AI应用中心"可能是二级部门，LLM 不应猜测它是三级部门而拒答，应直接调用工具。

3. **"/"分隔的部门名称必须理解为层级关系**：当用户输入"A/B/C"格式（用"/"明确分隔三个层级）时，理解为"一级部门/二级部门/三级部门"的层级关系。例如"人力资源部/中国区/COE"表示三级部门"COE"。**但即使如此，也应将完整名称传给工具处理**，由工具返回具体的拒绝原因或候选列表。

4. **部门找不到时的处理**：
   - **部门名称未找到**（包括一级+二级组合中任一未找到、用户只提了一个部门名称但找不到）：直接输出工具返回的拒答信息，**不做重试，不自行替换为其他部门名称**。

5. **部门匹配到多个候选时的处理**：工具返回 `{"error": "...", "candidates": [...]}` 格式时，**必须将候选列表展示给用户，让用户自行选择，禁止自动选择任何候选部门**。即使候选列表只有 1 个部门，也必须告知用户并等待确认，不得自行使用该候选重新调用工具。输出格式示例："'信息技术部'匹配到以下部门，请确认您要查询哪个：\n1. 集团信息技术部（一级部门）\n（已排除2个三级及以下部门，成本报告仅支持一级/二级部门）"

6. **职级条件忽略**：`cost_report` 工具不支持按职级筛选成本。若用户提问包含职级条件（如"中国区19+的成本报告"、"20级以上2月份的成本"），应忽略职级条件，直接调用 `cost_report(dept_name="中国区", ...)` 查询该部门整体成本报告。

7. **月份自动校正**：工具会自动查询最新成本更新月份。若用户未指定月份，工具自动使用最新成本更新月；若用户指定的月份超过最新成本更新月，工具会自动回退到最新成本更新月生成报告（不会拒答）。LLM 无需做月份校验，直接将用户指定的月份原样传给工具即可。

8. **集团级查询**：支持"小米集团"、"小米公司"作为部门名称，查询集团整体成本数据。

### ⚠️ 多轮对话隔离（必须遵守）

**每次成本报告请求必须独立处理，严禁混入对话历史中其他查询的数据或结论。**

- 填充分析解读时，**只使用当前工具返回的表格数据**，禁止引用对话历史中其他部门/其他查询的数据
- 如果用户连续查询了多个部门（如先查"中国区"再查"汽车部"），每次报告的分析解读必须**完全独立**，仅基于当次工具返回的数据
- **同一查询重复提问必须结果一致**：对于相同部门、相同月份的成本报告请求，无论是首次查询还是重复查询，工具返回的数据相同时，LLM 填充的分析解读结论也必须保持一致。禁止因上下文中已有的其他查询结果干扰当前报告的分析结论。如果对话上下文中已有该部门的报告，重新生成时必须**完全忽略之前的分析内容**，仅基于当次工具返回的数据独立填充

### ⚠️ 成本报告相关追问处理（必须遵守）

用户提问与成本报告相关、但并非直接要求生成报告时（如"XX部门各子部门实际&预测成本详情"），按以下规则处理：

1. **前文已有该部门报告（上下文可见报告内容）**：可以基于已生成的报告内容作答。**数据必须与报告内容完全一致，严禁篡改任何数字或结论。**
2. **前文没有该部门报告**：**直接拒答**，回复："暂无该部门的成本报告数据，请先查询该部门的成本报告。"
3. **严禁先生成报告再利用报告内容回答问题**——如果用户没有主动要求生成报告，不得自行调用 `cost_report` 工具生成报告后再从中提取数据作答。

### ⚠️ 权限拒答后禁止自行替换部门（必须遵守）

- 如果工具返回"没有查到您XX部门的权限"或"没有查看XX部门成本报告的权限"，**直接输出该拒绝信息，立即停止**
- **禁止**在收到权限拒绝后自行替换为其他部门名称再次调用 `cost_report`
- **禁止**猜测或编造部门名称调用工具
- **禁止编造部门映射关系**：输出话术中不得出现"XX是YY的默认映射部门"、"XX默认映射为YY"等编造的部门映射描述。系统中不存在任何部门间的"默认映射"机制，各部门名称独立使用，严禁自行臆造关联

### ⚠️ 多部门查询的权限处理（必须遵守）

**场景一：并列查询多个部门**（如"A和B的成本报告"）

- 需要**逐个调用** `cost_report`
- **只要其中任何一个部门返回权限拒绝，立即停止所有后续调用**，直接输出权限拒绝信息，已成功生成的报告也**不发送**
- 拒绝信息应明确列出哪些部门没有权限（如"没有查看B部门成本报告的权限"），多个部门无权限则逐一列出
- 禁止"跳过无权限部门，只展示有权限部门"的做法——要么全部有权限才输出，否则整体拒答

**场景二：查询某部门下各子部门对比**（如"C部门下各二级部门的成本报告对比"）

- **优先检查父部门权限**：先调用 `cost_report(dept_name="C")` 检查 C 部门是否有权限。如果 C 部门无权限，**直接拒答**，不再查询任何子部门
- **父部门有权限后，逐一查询子部门**：对 C 下每个二级部门调用 `cost_report`，如果某个子部门返回权限拒绝，该子部门报告不生成，输出该子部门的权限拒绝信息

### 不生成报告的情况（拒答优先级最高）

工具在生成报告前会自动执行**前置资格检查**，**该检查优先于权限校验**——即使用户无权限，若部门本身不满足资格条件，工具也会优先返回资格检查的拒绝原因。当以下任一条件不满足时，工具返回拒绝原因（纯文本，非报告骨架），**LLM 应直接输出该拒绝原因，不要尝试自行编写 SQL 或生成报告**。

| 检查项 | 拒绝条件 | 工具返回的拒绝原因 |
|---|---|---|
| 集团总办 | 部门属于集团总办（dept_id_lvl1='PO'） | 集团总办不生成报告 |
| 集团总干部部 | 部门属于集团总干部部（dept_id_lvl1='OD'） | 集团总干部部不生成报告 |
| 顾问群 | 部门属于顾问群（dept_id_lvl1='CG'） | 顾问群不生成报告 |
| 部门未成立 | 部门生效日期晚于查询月份 | 因部门尚未正式成立（成立时间为XXXXXXXX），不满足报告生成条件 |
| 人数不足 | 该月在职人数少于 5 人 | 该部门在职人数少于5人，为保护用户薪酬隐私，不满足报告生成条件 |
| 未编制预算 | 该年度预算金额为 0 | 该部门成立时间XXXXXXXX，该部门未编制预算，不满足报告生成条件 |
| 无成本数据 | 该月份无任何成本数据记录 | 该部门在该月份无成本数据，不满足报告生成条件 |

**LLM 处理规则：**
- 工具返回格式为 `「部门名」X月成本报告无法生成：{拒绝原因}`
- LLM **直接原样输出**该提示，不做额外修饰、不尝试补充分析
- **禁止**在收到拒绝原因后自行调用 `sql_query` 或其他工具尝试绕过检查
- 集团级查询（小米集团/小米公司）跳过上述所有检查，始终生成报告

## 二、错误处理（必须遵守）

如果 `cost_report` 工具返回 `{"error": ...}`（包括超时、查询失败等任何错误），**必须再次调用 `cost_report` 工具重试**（最多重试 1 次）。

**绝对禁止**：
- ❌ 在工具返回错误后自行编写 SQL 查询
- ❌ 自行拼凑报告数据
- ❌ 调用 `sql_query` 替代

如果重试仍然失败，向用户说明"成本报告工具暂时繁忙，请稍后再试"，不得自行生成替代报告。

## 三、工具返回值

工具有三种返回情况：

### 1. 成功

返回 **JSON**，包含两个字段：
- `sections`：7 个分段（summary, q1, q2, q3_1, q3_2, q3_3, q4），每段含 `key`（段落标识）、`title`（飞书卡片标题）、`content`（Markdown 内容）
- `full_report`：完整 Markdown 报告（所有 sections 拼接，用于飞书文档保存场景）
- 每个 section 的 `content` 中包含已填充数据的表格、核心结论（已有具体数字）、以及占位提示（括号包裹的说明文字，需 LLM 填充）

**工具负责数据查询、表格渲染和核心结论数据填充。分析解读由 LLM 根据表格数据生成。**

### 2. 不满足生成条件

返回纯文本拒绝原因（如 `「企业智能协同部」2月成本报告无法生成：因部门尚未正式成立...`）。LLM **直接输出**，不做额外处理。

### 3. 错误

返回 JSON：
```json
{"error": "成本报告生成失败: ..."}
```

## 四、LLM 输出规则

> **⚠️⚠️⚠️ 最高优先级规则：成本报告必须以 7 张独立飞书卡片的形式逐张发送，严禁合并为一张卡片或一条文字消息！**
>
> 工具返回 JSON，`sections` 包含 7 个段落。LLM 必须对每个段落：填充占位提示 → 组装为独立的飞书卡片 JSON → 通过 `exec` 工具逐张发送。
> **每个 section 对应一张独立卡片，共 7 张，必须发送 7 次 `exec` 调用。**
> 表格数据保持原样，结论和分析由 LLM 根据表格中的具体数字生成。

### 完整输出铁律（最高优先级，违反视为严重错误）

每个 section 的内容必须**完整、逐字输出**，禁止以任何理由截断、省略或缩减：

1. **表格必须完整输出所有行和列**——工具返回多少行就输出多少行，禁止只输出"前N行"或用"……"省略剩余行。即使表格很长（如月度明细 12 行、职级分布 8+ 行），也必须全部输出
2. **7 个 section 必须全部发送**——禁止因响应长度压力而跳过任何 section
3. **每个 section 内的所有表格必须全部输出**——禁止省略
4. **整体结论 + 各 section 分析解读必须全部填充**——禁止只输出表格不写分析，也禁止只写分析不输出表格
5. **禁止合并或简化表格**——不得将多张表合并为一张，不得删除"看起来重复"的列

如果担心输出过长，仍然必须完整输出——成本报告的完整性优先于长度控制。

### 🔴 红线规则（输出前必须逐条校验，违反视为严重错误）

以下红线适用于整篇报告所有「整体结论」和「分析解读」模块，**优先级最高**，任何分析解读在输出前必须逐条对照。

**红线1：禁止跨维度人均对比推论**
不同群体（正式vs外包、不同城市/区域）职级结构不同，禁止输出跨群体人均绝对值对比、倍数对比或"成本优势/差异"类结论。
- ❌ "正式员工人均是外包的X倍，外包成本优势显著"
- ❌ "北京人均（X万）是武汉（Y万）的N倍，区域差异大"
- ❌ "北京人均4.51万显著高于武汉1.84万"
- ❌ "外包人均远低于正式员工"、"A地区成本远高于B地区"
- ✅ 各群体只展示自身的同比/环比变化趋势

**红线2：Zone区禁止展示金额**
Zone区只展示人数分布，禁止任何金额数字。
- ❌ Zone 1（28.56万）、现金差3.44万、总年薪差5.37万
- ✅ Zone 1: 存量X人、新进Y人、离职Z人

**红线3：人均成本禁止「绝对值+定语」**
人均成本只允许展示整体同比/环比，带任何定语（城市/区域/员工类型/职级等）时禁止展示绝对值。
- ❌ "北京人均成本26.65万"、"外包人均3.10万"、"18级人均22.18万"
- ❌ "武汉人均3.34万，同比+49.8%"（绝对值+同比同时出现也禁止）
- ✅ "人均成本同比+5.3%"
- ✅ "武汉人均成本同比+49.8%"（只写同比，不写绝对值）

**红线4：禁止篡改工具返回的任何数据（最高优先级）**
`cost_report` 工具返回的所有内容（表格、核心结论、数字、文字）均由 SQL 查询直接渲染，是**唯一权威数据源**。LLM **必须逐字原样输出**，严禁任何形式的篡改、重新计算或重新格式化：
- ❌ 修改表格中的任何数字（如把"1,234.56"改成"1234.6"、四舍五入、重新计算）
- ❌ 修改表格中的任何文字（如把"总成本"改成"总计"、把"现金薪酬"改成"现金"）
- ❌ 增删表格行（如删除"看起来多余"的汇总行，或自行添加合计行）
- ❌ 增删表格列（如删除"去年同期"列，或添加"差额"列）
- ❌ 调整表格行的排列顺序
- ❌ 修改表头文字（如把"成本（万）"改为"金额（万元）"）
- ❌ 将表格数字的格式改变（如去掉千分位逗号、改变小数位数、去掉百分号）
- ❌ 修改核心结论中的数字（如把工具返回的"同比+5.3%"改成"同比+5%"）
- ❌ 对工具返回的数字进行二次计算后输出（如自行用两个数相除得出百分比）
- ✅ 唯一允许的操作：原封不动复制工具返回的内容到飞书卡片

**违反此红线视为严重错误**——表格数据的准确性直接影响管理决策，任何篡改都可能导致错误判断。LLM 的职责是根据表格数据撰写「分析解读」，而非修改数据本身。
**判断标准**：如果你输出的任何数字与工具返回的原始数据不一致，即视为篡改。当你在分析解读中引用数据时，必须直接复制工具返回的原始数字，禁止自行计算或推导新数字。

**红线5：禁止单位转换——所有金额必须以"万"为单位**
工具返回的金额单位统一为"万"，LLM 在分析解读中引用金额时**必须保持"万"为单位**，严禁自行换算为其他单位。
- ❌ "总成本约X万万"、"总成本约X亿"、"累计成本达到X亿元"
- ❌ 将"12,345.67万"换算为"12.35亿"或"1.23亿"
- ❌ 将"10,000万"写成"1亿"
- ✅ 直接使用工具返回的原始数字+万：如"总成本12,345.67万"、"累计成本10,000万"

**红线6：禁止自动生成图表**
LLM 严禁在报告中插入任何图表、图形、可视化元素。报告内容必须严格按照工具返回的表格+文字分析格式输出，不得额外添加。

- ❌ 使用 ASCII art 绘制图表
- ❌ 添加工具返回内容中不存在的任何可视化元素
- ✅ 仅输出工具返回的表格（Markdown table）+ LLM 填充的文字分析

**红线7：数值精度必须与工具返回的原始数据完全一致**
LLM 在整体结论、分析解读中引用任何数值时，**必须保持工具返回的原始精度**，严禁四舍五入、截断小数位或增加小数位。同一个数值在报告不同位置出现时，精度必须前后一致。
- ❌ 工具返回"同比+5.3%"，整体结论中写成"同比+5%"（丢失小数位）
- ❌ 工具返回"12,345.67万"，分析解读中写成"12,346万"（四舍五入）

**红线8：禁止高职级成本对比分析**
整体结论和分析解读中，禁止出现针对高职级（如19级、20级及以上）的成本对比、成本占比、成本增速等分析。职级维度的人均成本表格由工具原样输出即可，LLM 在撰写分析时**不得单独提及或对比高职级的成本数据**。
- ❌ "19级及以上人均成本同比+X%，增速显著"
- ❌ "高职级员工成本占比达X%，同比上升"
- ❌ "20级人均成本X万，远高于其他职级"
- ❌ "高职级成本增长是整体成本上升的主要驱动因素"
- ✅ 分析解读聚焦整体人均成本的同比/环比变化趋势
- ✅ 如需提及职级维度，仅描述整体职级结构变化趋势（如"职级结构整体稳定"）

**红线9：禁止基于「时间进度」得出预算相关结论**
判断预算执行情况（可控/偏紧/超支风险）时，**只能基于「全年预估使用率 vs 100%」**，禁止用「时间进度」作为参照基准。
原因：预算使用率（= 累计实际 ÷ 全年预算）与时间进度（= N ÷ 12）分母不同，且成本按月分布不均匀（年终奖、股权归属等集中在特定月份），用时间进度推断预算快慢没有业务意义。
时间进度可以作为背景信息单独提及，但**禁止与预算使用率关联推导结论**。
- ❌ "预算使用率91.3%，高于时间进度33.3%，支出节奏偏快"
- ❌ "预算使用率94.2%，远超时间进度33.3%"
- ❌ "低于时间进度对应的预期水平，预算整体可控"
- ❌ "预算执行快于/慢于时间进度"
- ✅ "全年预估使用率93.7%，总量可控"（基于全年预估 < 100% 得出可控结论）
- ✅ "全年预估使用率105%，节奏偏紧，有超预算风险"（基于全年预估 > 100% 得出偏紧结论）
- ✅ "预算使用率91.3%，同比+X%"（单独描述预算使用率变化趋势）
- ✅ "报告期为1-4月，时间进度33.3%"（作为背景信息单独提及，不关联预算结论）

## 五、飞书卡片发送流程

> **⚠️ 成本报告必须使用 `msg_type=interactive` 发送飞书交互卡片，严禁使用 `msg_type=text` 或其他类型。**

### 发送步骤

对工具返回的 `sections` 数组（共 7 段），**按顺序逐个执行**以下操作：

1. **读取** section 的 `content`（Markdown 内容）
2. **填充**占位提示为实际分析文字（遵循上述红线和填充规范，**禁止篡改任何数据、禁止改变数值精度**——表格中两位小数就写两位小数，严禁缩减为一位或整数）
3. **组装**飞书卡片 JSON，使用 `exec` 工具调用 `send_card()` 函数发送（参考 [references/report/send-card.md](references/report/send-card.md)）
4. `receive_id` **必须使用 runtime context 中的 `Chat ID`**（不是 `sender_open_id`）

### 飞书卡片 JSON 结构

```json
{
  "schema": "2.0",
  "header": {
    "title": {"tag": "plain_text", "content": "SECTION_TITLE"},
    "template": "purple"
  },
  "body": {
    "elements": [
      {"tag": "markdown", "content": "FILLED_CONTENT"}
    ]
  }
}
```

> **🚨🚨🚨 `FILLED_CONTENT` 必须是填充分析后的完整 Markdown 文本，不能为空！**
>
> - `FILLED_CONTENT` 就是 section 的 `content` 字段经过"填充占位提示"后的完整 Markdown 字符串，包含核心结论、表格、分析解读等全部内容
> - **卡片内容为空 = 严重错误**：如果 `body.elements[0].content` 为空字符串 `""`、为 `"FILLED_CONTENT"` 占位符原文、或未赋值，飞书卡片会显示为空白
> - **卡片显示 JSON 原文 = 严重错误**：如果把卡片 JSON 结构或 sections 数组放进了 `content` 字段，用户会看到 JSON 乱码而非报告内容
> - 正确做法：读取 section 的 `content` → 将其中的占位提示替换为分析文字 → 将替换后的**完整 Markdown 字符串**放入 `body.elements[0].content`
>
> 详细调用示例见 [references/report/send-card.md](references/report/send-card.md)

**字段说明：**
- `SECTION_TITLE` = section 的 `title` 字段（直接使用，不修改）
- `FILLED_CONTENT` = section 的 `content` 经填充占位提示后的完整 Markdown 文本（包含核心结论 + 表格 + 分析解读，不是 JSON）
- **同一份报告的 7 张卡片颜色必须一致**：每次成本报告的 7 张卡片 `header.template` 必须使用相同的颜色值。可以从 `purple`、`blue`、`indigo`、`violet`、`turquoise` 中任选一个，但一旦选定，7 张卡片必须全部使用该颜色，禁止出现同一份报告中卡片颜色不统一的情况。
- **卡片标题必须包含【官方精选报告】前缀**：工具返回的 summary section 的 `title` 字段包含「【官方精选报告】」前缀，LLM 在组装飞书卡片时**必须原样保留**该前缀，严禁删除、修改或省略。发送前检查 summary 卡片的 `SECTION_TITLE` 是否包含「【官方精选报告】」，缺失则视为严重错误
- **卡片标题不要在内容中重复**：`header.title` 已经是卡片标题，`FILLED_CONTENT` 中不要再以 `# 标题` 或加粗文字等形式重复一遍标题
- **换行符必须保留**：`FILLED_CONTENT` 中的换行必须使用 `\n`（JSON 转义的换行符）。工具返回的 `content` 本身包含 `\n` 分隔各段落、表格行和分析条目，组装 JSON 时必须原样保留这些 `\n`，严禁将多行内容拼接成一行。如果换行丢失，飞书卡片将无法正确渲染 Markdown 格式（表格错乱、段落合并）

### 计算口径说明

- **当月区域人均成本**：采用简单公式 `当月成本 / HC数 / 10000`
