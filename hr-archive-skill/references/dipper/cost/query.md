# 人力资源成本北斗查询技能（hr-cost-query）

## 一句话定位

面向 **HR 成本 / 预算 / 实际 / HC** 提问，在北斗（`bizId=119`）**双聚合模型**（成本金额 + HC，见 `measures_catalog.md` 模型选型）上取数并呈现结果。**不处理**其他业务线指标。

| 参考文档 | 用途 |
|----------|------|
| `references/dipper/cost/measures_catalog.md` | 指标/维度字典、口语速查、拒答与特殊问法（§一～§六） |
| `references/dipper/cost/enum.md` | 员工类型、区域等筛选枚举 |
| `references/dipper/dipper-api.md` | MCP 工具、`Condition` 与 `data_query` 请求体示例 |

---

## ‼️ 强制约束：callContext

**每次 MCP 调用必须携带 `callContext`，缺一不可。**

```json
{
  "callContext": {
    "messageId": "${message_id}",
    "userInput": "${用户原始输入}",
    "userSkill": "${skill_name}"
  }
}
```

| 字段 | 说明 |
|------|------|
| `messageId` | 飞书 `message_id` |
| `userInput` | 用户原文，不翻译、不摘要；多条补充用 `\n` 拼接 |
| `userSkill` | 直接调用填 `hr-cost-query`；父技能包装时填**父技能名** |

判断场景：有父技能透传的 `_PARENT_SKILL` / `_USER_INPUT` / `_MESSAGE_ID` 则用透传值，否则 `userSkill=hr-cost-query` 且从当前会话取另两字段。

> 下文 **Step 1～3** 与 callContext 场景无关，仅填充规则不同。

## 固定取数参数（硬编码）

| 项 | 值 |
|----|-----|
| `bizId` | **119**（禁止询问业务线、禁止其他 bizId） |
| `type` | **1**（聚合，默认路径） |
| `queryKey` | 按 `measures_catalog.md` **模型选型**：成本金额 **`a3fe268e90104b459919c9f0c95dfbf6`**；HC **`d1f192be8da24cc2b0877689551e18c8`**（禁止臆造、禁止跨模型混用 alias、禁止采纳 `data_search_measures` 返回的其他 key） |
| 工具 | 主路径仅 **`data_query`**；辅助见下节 |

**指标映射**：先查 **`measures_catalog.md`**（§五拒答 → **模型选型** → 对应模型指标/维度表 → §四 口语）。`calculateType=4` 组合指标（如 `ai_qa_cost_rate`、`ai_qa_est_hc_rate`）**直接作 `measures`**，勿拆原子手算；同比/环比/占比见 catalog **§六**。

**下钻**：在同一 `type=1` 请求中增加更细 `dimensions`（如 `dept_id_lvl3`～`dept_id_lvl5`）与 `condition`，**不**切换明细模型或 `data_measure_detail_query`。

## 关键口径：`mon`

用户说的「× 月」= **`condition` 里约束 `mon`**（`YYYYMM01`，如 2026 年 4 月 → `"20260401"`），**不得**用 `date` 代替业务归属月。

| 步骤 | 规则 |
|------|------|
| `mon` | 单月在 `condition` 用 `filterType: 1`(EQ);多月用 `filterType: 10`(INCLUDE)+ 字符串数组 |
| 时间默认 | 用户未指定任何月份时：<br>• **HC 类指标**（`hc`/`_hc_` 字段，如 `ai_qa_est_hc`、`ai_qa_use_rate`） → 默认 `mon = 当前自然月`（如今天 2026-06-04 → `"20260601"`，`filterType: 1` EQ）；HC 类**不发 Step 0**（与实际成本入库节奏不同）。<br>• **非 HC 类**（金额/成本/比率/使用率等） → **必须先发 Step 0** 获取 `max_cost_mon`，默认 `mon ∈ [当年1月, max_cost_mon]`（详见下方「Step 0 实际数据边界探查」）；Step 0 失败时降级为 `[当年1月, 当前月]` + 「部分月缺数」兜底收敛。 |
| 部门 | 组织维度含 `dept_id_lvl0`～`dept_id_lvl5` 与 `dept_nm_lvl0`～`dept_nm_lvl5`；解析方法见下方「**部门参数解析（两步法）**」 |

**`Condition` 形态**（禁止 `relation`/`conditions` 等非北斗结构）：根为 `{ "operator": "AND"|"OR", "subConditions": [...] }` 或单叶子 `{ "field", "filterType", "value" }`。常用 `filterType`：`1`=EQ，`10`=INCLUDE。完整见 `references/dipper/dipper-api.md`。

多月 + 部门示例：

```json
{
  "operator": "AND",
  "subConditions": [
    { "field": "dept_id_lvl1", "filterType": 1, "value": "DEPT_ID_PLACEHOLDER" },
    { "field": "mon", "filterType": 10, "value": ["20260101", "20260201", "20260301"] }
  ]
}
```

## Step 0：实际成本边界探查（非 HC 类多月查询前置）

**目的**：避免把"实际成本未入库的月份"带入 condition，引发使用率/累计口径失真。

> Step 0 与 Step 1（部门向量检索）**互不依赖、平级前置**：可并行发起；"Step 0" 编号不代表优先级最高。**不缓存**——同一会话内每次成本查询都重新发。

### 触发条件

| 场景 | 触发 | 用途 |
|------|------|------|
| 任何含 `ai_qa_cost_amt` 的多月查询 | ✅ **必发** | 用 `max_cost_mon` 裁剪 mon 上限 |
| 双表场景（含「预测/预计」语义） | ✅ **必发** | `max_cost_mon` 作为 N-1 切分点；附带读 `y_fcst_desc` 用于预测段说明文案 |
| 单月查询（用户明确指定某个具体过去月） | ❌ 不发 | 无累计/裁剪需求 |
| HC 类查询 | ❌ 不发 | HC 与成本入库节奏不同，沿用 HC 默认（当前自然月） |
| 用户显式指定时间范围（含未来月） | ✅ 发但**不裁剪用户原文** | 仅在 `meta.warnings` 标注「>max_cost_mon 的月份为预算/预测口径」 |

### 请求体

双表场景额外取 `y_fcst_desc` 作预测段说明文案（不进主表 dimensions）：`dimensions: ["max_cost_mon", "y_fcst_desc"]`。

```json
{
  "type": 2,
  "bizId": 119,
  "queryKey": "05b5f2a1749c4925bc46955e9776f9b4",
  "measures": [],
  "dimensions": ["max_cost_mon"]
}
```

### 失败降级

| 情况 | 处理 |
|------|------|
| 200 但 `max_cost_mon` 为空 | `meta.warnings` 标注；降级走 `[当年1月, 当前月]` + 缺数收敛兜底 |
| 403 | 同上，降级；不阻断主查询 |
| 500/超时 | 重试 1 次仍失败 → 降级，`meta.warnings` 标注「未取得最新实际月份，时间范围按日历推断」 |

### 衔接

- 主查询 + 二次累计 `mon` 上限 **≤ `max_cost_mon`**（用户显式月份除外）
- 双表 `N = max_cost_mon + 1`
- 缺数收敛降级为兜底，仅应对 Step 0 失败 / 极端缺数

## 部门参数解析（两步法：识别与取数分离）

**铁律**：主查询 `data_query` 的 `condition` 中部门字段**只允许 EQ**，禁止 LIKE。LIKE 会跨部门静默求和（例：`dept_nm_lvl*` 用 `filterType=12` 模糊匹配 会把所有名称含「手机部」的部门数据合并，且 `dimensions` 不带 `dept_*` 时无法察觉）。

### 流程

```
用户原文 → [Step 1 识别] 唯一规范部门名 + 层级 → [Step 2 取数] EQ
```

### Step 1：识别（输出唯一 dept_id + 完整规范名 + 层级）

**所有部门查询必须经 Step 1 向量检索取得唯一 `dept_id_lvl{N}`**，不设单步豁免。即使用户原文已给「lvl2 新业务部」这类完整明示，仍走 Step 1——避免同名异 ID 的部门（如多个 lvl2「新业务部」）在 Step 2 EQ 名称查询时被静默聚合。

#### 字段命名映射

向量检索响应使用**新格式** `dept_lvl{N}_id` / `dept_lvl{N}_name`，主查询 `data_query.condition.field` 使用**旧格式** `dept_id_lvl{N}` / `dept_nm_lvl{N}`。两者等价、模型同时接受；写入主查询时统一用旧格式 `dept_id_lvl{N}` 保持文档一致。

#### 向量检索：`data_search_dimension_values` 关键词模式

单次调用即可跨所有层级检索（无需 lvl1→lvl2 逐级试探）：

```json
{
  "bizId": 119,
  "keyword": "<用户原文核心词原始字符串>",
  "pageNum": 1,
  "pageSize": 10,
  "callContext": { ... }
}
```

**关键词使用约束**（适用 condition 所有维度筛选：部门 / 序列 `job_fml_frs_nm` / 城市 `loc_cty_desc` 等）：`keyword` 必须是**用户原文核心词原始字符串**，**禁止自行截断或去后缀**。

- ✅ 「湖北分公司」→ `"湖北分公司"` / 「研发序列」→ `"研发序列"` / 「集团信息技术部」→ `"集团信息技术部"`
- ❌ → `"湖北"` / → `"研发"` / → `"信息技术"`（自行去后缀或简写）

#### 候选处理流程

响应 `data.data[]` 已按 `score` desc 排序，每行含 `value`（部门完整规范名）/ `dimensionAlias`（如 `dept_lvl2_name`）/ `joinIdAlias`（如 `dept_lvl2_id`）/ `joinIdValue`（部门 ID）。按以下顺序处理：

1. **按主查询 queryKey 过滤层级**——cost 模型（成本 queryKey `a3fe268e90104b459919c9f0c95dfbf6`）保留 `joinIdAlias ∈ {dept_lvl1_id, dept_lvl2_id}`；HC 模型（HC queryKey `d1f192be8da24cc2b0877689551e18c8`）保留 `joinIdAlias ∈ {dept_lvl1_id ... dept_lvl5_id}`。两者均**丢弃 lvl0**（lvl0=MI 由「集团/小米公司」特殊规则节专门处理，非向量检索路径）。
2. **去重**：按 `(joinIdValue, joinIdAlias)` 元组去重——同一部门同时返回 ID 与名称两字段视为同一候选。
3. **判定**：
   - **0 个候选** → 走「命中数量兜底」0 行话术。
   - **完全匹配 = 1**（候选 `value` 与用户原文核心词完全一致）→ 取该候选 `(joinIdValue, value, 层级)` 直进 Step 2。
   - **完全匹配 ≥ 2**（同名异 ID 或多个全等候选）→ 列出**全部完全匹配候选**反问，禁止补 top3、禁止自行选第一条。
   - **完全匹配 = 0 但有候选**（用户原文是简写、误写、别称）→ 列出 top3 反问。

**反问候选格式**：`找到 N 个候选：① {value} (ID: {joinIdValue}, lvl{N}) ② ...，请确认要查询哪一个`。同名异 ID 时附带「隶属 {上级部门名}」（响应未带上级名时可省略）。

> - cost 模型 lvl3+ 已被 `measures_catalog.md` §五红线拦截，过滤层级时一并丢弃；HC 模型无此限制。

### Step 2：取数

`data_query` 的 `condition` **优先用** `dept_id_lvl{N}` + `filterType=1`（EQ）+ Step 1 拿到的 `dept_id`（避免同名异 ID 静默聚合）；仅当 Step 1 因接口异常未返回 `dept_id` 时，退回 `dept_nm_lvl{N}` EQ + 完整规范名（同时在 `warnings` 标注「未取得 dept_id，按名称查询，重名部门数据可能被合并」）。

**dimensions 默认不加部门字段**：
- 只查单部门的月度趋势 → `dimensions: ["mon"]`，**不加** `dept_nm_lvl{N}` / `dept_id_lvl{N}`
- 用户要求子部门下钻（如"手机部各二级部门成本"）→ 加 `dept_nm_lvl{N+1}` 或 `dept_id_lvl{N+1}`
- **特例**：手机部排除新业务部（NEQ 过滤）等特殊规则要求 dimensions 含被过滤字段 → 按特殊规则节执行，不适用本默认

### 0 候选话术

Step 1「候选处理流程」中"0 个候选"分支统一话术：明确告知用户「**未找到名称含「XX」的部门**，请确认部门名称或提供更完整的全称」。**禁止**伪装成「该部门暂无成本数据」——必须区分「部门不存在」与「部门存在但无数据」。

完全匹配 = 1 / ≥2 / top3 反问的处理逻辑见 Step 1「候选处理流程」表格，本节不重复。

### 禁止行为

- ❌ 主查询 `data_query` 的 `condition` 写 LIKE（跨部门静默求和）
- ❌ 把「未找到部门」模糊回答为「暂无数据 / 0 元」
- ❌ Step 1 完全匹配 ≥2 行时自行聚合或选第一条；同名异 ID 必须列出反问
- ❌ 跳过 Step 1 直接用名称 EQ 取数（即使用户明示了「lvl2 XX 部」）
- ❌ 跳过指标类型分流过滤——cost queryKey 不丢弃 lvl3+ 候选会触发 §五 红线
- ❌ 跳过去重直接选 D[0]——`(joinIdValue, joinIdAlias)` 完全相同的多行视为同一候选
- ❌ 截断/简写 `keyword`（如把「研发序列」写成「研发」），见 Step 1「关键词使用约束」

## 部门特殊过滤规则

> **适用范围**：本节所有规则对 cost 通道下的**成本模型**与 **HC 模型**同样生效，与 measures 类型（金额/HC/比率）无关。规则触发条件仅看部门字段（`dept_nm_lvl{N}` / `dept_id_lvl{N}`），不看指标类型。

### 集团 / 小米公司（lvl0）→ `dept_id_lvl0='MI'`

**触发**：用户原文出现「集团」「小米公司」「全公司」「整个集团」「全集团」「公司」（指代全公司语境，非业务部门名）→ **跳过部门参数解析两步法**，直接按全公司层级取数。

**不触发**：
- 仅"小米"两字（容易误命中"小米手机部"等业务部门）
- 与具体业务部门名同时出现（如"小米公司手机部"应视为询问手机部）
- 「**归属**集团成本」「**部门汇总成本**」→ 走「特殊指标」节规则

**Condition**：`dept_id_lvl0 EQ 'MI'`

**Dimensions**：按 Step 2 默认规则（不下钻只加 `mon`）。

### 手机部（lvl1）→ 自动排除新业务部（dept_id_lvl2=HW38）

**触发**：Step 1 识别结果为 `dept_nm_lvl1 = 手机部` 或 `dept_id_lvl1 = HW`；**与所问指标类型无关**（金额/预算/HC/使用率/结余 等任一指标都适用）。

**Condition 追加**：
- `{ "field": "dept_id_lvl2", "filterType": 2, "value": "HW38" }`（NE，不等于）
- 同时在 `dimensions` 加入 `dept_id_lvl2`（满足后端"condition 字段必须在 dimensions 中"约束）

**渲染**：
- 用户问汇总：客户端按月聚合 lvl2 行得到手机部总额（已自动剔除 HW38），**不展示** lvl2 拆分
- 用户问 lvl2 明细：直接展示，无新业务部行
- **标题与 `summary` 中部门名展示为「手机部 (除新业务部)」**（用户直接查 lvl2=新业务部 时不触发本规则，标题正常展示「新业务部」）
- `warnings` 加一条：「已按规则剔除"新业务部"（dept_id_lvl2=HW38）」

**强制性**：本规则不可由用户原文措辞覆盖。即使用户写"包含新业务部 / 不要剔除 / 含 HW38"等，仍按规则剔除。如用户需要新业务部数据，应直接查询新业务部（Step 1 命中 lvl2 = 新业务部 → 不触发本规则）。

**不触发**：
- 用户直接查 `lvl2 = 新业务部`（Step 1 命中 lvl2 = 新业务部）→ 正常返回新业务部数据
- 用户查上级（lvl0/集团）→ 不应用（手机部规则只在 lvl1 边界生效）

### 归属集团成本 / 部门汇总成本（特殊指标）

**触发**：用户原文出现「**归属集团成本**」或「**部门汇总成本**」（关键字精确匹配）→ 进入本规则，**跳过部门参数解析两步法**（这两个指标按固定层级取值聚合，不依赖用户给出的部门名）。

**指标定义**：

| 指标 | 计算逻辑 |
|------|---------|
| 归属集团成本 | `ai_qa_cost_amt` where `dept_id_lvl1 = 'CORP'` |
| 部门汇总成本 | `(dept_id_lvl0='MI' 的 ai_qa_cost_amt) − (dept_id_lvl1='CORP' 的 ai_qa_cost_amt)` |

> MI 是公司层级（lvl0，含全公司），CORP 是 lvl1（集团总部），部门汇总成本 = 全公司 − 集团总部，必须两次查询后客户端按月相减。

**查询构造**：

- **归属集团成本**（单查询）：`dimensions=["dept_id_lvl1","mon"]` + `condition: dept_id_lvl1 EQ 'CORP'` + `mon ...`
- **部门汇总成本**（两次查询，客户端按月相减）：
  - 查询 A：`dimensions=["dept_id_lvl0","mon"]` + `condition: dept_id_lvl0 EQ 'MI'` + `mon ...`
  - 查询 B：同归属集团成本
  - 合计：`部门汇总成本[mon] = A[mon] − B[mon]`

**展示**：
- 用户同时问两个指标 → **合法**，一并返回（两指标各自独立查询、独立展示）
- **部门汇总成本只展示合计**，不展示 MI、CORP 两段明细

**混搭部门拒答**：用户把这两个指标与具体部门名混搭（如"手机部归属集团成本"、"信息技术部部门汇总成本"）→ 不发起查询，按「**归属集团成本/部门汇总成本按集团范围统计，不支持按部门拆分**」回复，禁止用部门数据替代。

## 工具（按优先级）

| 优先级 | 工具 | 何时使用 |
|--------|------|----------|
| 必用 | `data_query` | 默认 `type=1` + 固定 `queryKey`；非 HC 类多月/双表查询前置 **Step 0** 探查 `max_cost_mon`（`queryKey=05b5f2a1749c4925bc46955e9776f9b4`，`measures=[], dimensions=["max_cost_mon"]`）；多月查询另发一次"去 mon 维度"调用取累计合计 |
| 必用 | `data_search_dimension_values` | **Step 1 部门识别主路径**（关键词模式：`bizId=119` + `keyword=用户原文核心词`），返回 `(joinIdValue, joinIdAlias)` 即唯一 dept_id；其他维度枚举筛选值同样走此工具 |
| 按需 | `data_search_measures` | catalog 对不上口语时搜 alias（**不**用于选 queryKey） |
| 备用 | `data_query` `type=3` | **仅当** catalog §一 确认指标不在模型内，且聚合查询已失败时使用；仍 `bizId=119`，**不传** `queryKey`，请求体见 `references/dipper/dipper-api.md`「type=3 降级」。**禁止**为省事默认 type=3，或用 type=3 规避固定模型权限/口径问题 |
| 暂不纳入主流程 | teamquery、`data_measure_detail_query`、`data_module_meta`（每次调） | 人力域当前不用 |

## 工作流程（三步）

### Step 1：解析意图并构造请求体

1. **域校验**：非人力成本（如销量）→ 拒答，说明仅 `bizId=119`。
2. **拒答预判**（不发起查询）：见 `measures_catalog.md` **§五**（如四五级月度总预算、到人薪酬、集团总办/集团总干干部不支持查成本、三级及以下部门不支持成本查询）。
3. 解析 `measures`、`dimensions`、`condition`（含 `mon`、`dept_id_lvl*`）；对照 catalog §一、§四 选 alias。
4. **复合问法**：优先 catalog §1.2 的 `ai_qa_*` 组合指标；仅自定义口径时再多次 `data_query` + Agent 侧合成（`warnings` 写明公式）。
5. 信息不足 → 追问（≤2 轮，一次一个问题或 A/B/C）；仍不足 → `status=failed`。
6. 组装 **`request_body`**：`type=1`、`bizId=119`、固定 `queryKey`、`measures`、`dimensions`、`condition`（含 `mon`，业务时间口径来源）、`pageParam`、`callContext`。

**指标未知时**：`data_search_measures`（`bizId=119`）搜 alias，重试 1 次；仍失败则追问，**不**换 queryKey。

---

### Step 2：执行查询

```bash
mcporter call data-query.data_query --args '{request_body}' --output json
```

- MCP 超时 → 重试 1 次（间隔 2s）；再失败 → `status=failed`，输出失败诊断包（见下）。
- `response.code !== 200` → 修正 `measures`/`condition` 后重试 1 次；**禁止**通过更换 queryKey 绕过。
- 模型报错「指标不在模型」→ 对照 catalog §一；确认不在模型且允许时，走工具表「备用」行的 `type=3` 降级，**仅此一种降级**。

---

### Step 3：处理结果并呈现

1. `code=200`；检查 `data.total`、`noAuthMeasures`。
2. **过滤合计行**后再汇总，避免翻倍。
3. 按 Output Contract 格式化：**金额**按万元/亿元两档展示（2 位小数、四舍五入），**使用率** 2 位 + `%`，**人数/HC** 整数；`result` 内可保留接口原值（元）。默认按主指标降序。
4. 写入 `meta`：`bizId=119`、`type=1`、`queryKey` 固定值、`time_range`、`sql`/`trace`（来自 `extendInfo`）。
5. **轻量校验**（原 Verifier 合并于此）：精度与 `null_policy`、汇总与明细一致（±0.01）、`summary` 与数据一致、时间范围与 `condition.mon` 一致。
6. 按「输出格式」渲染；如有缺月，遵循下方「输出格式」标题与缺月规则，**禁止**使用 T+1/未发布等措辞解释月度缺数。

## Output Contract（摘要）

| 属性 | 值 |
|------|-----|
| `timezone` | `Asia/Shanghai` |
| `raw_unit` | 接口金额类指标原值为 **元**（`fieldType=8`，见 catalog §一） |
| `number_precision` | 见下表「展示精度」；**内部** `result.totals` / `items` 建议保留原值（元）便于校验 |
| `null_policy` | 缺失 `null`，空列表 `[]`，无数据 `total=0` |
| `dedupe_policy` | 过滤合计行后再计算 |

**展示精度**（面向用户的表格、`summary` 中的数值）：

| 指标类型 | 判定 | 展示规则 |
|----------|------|----------|
| **金额** | catalog `fieldType=8`，或名称含预算/成本/结余/金额等金额语义 | 对 **绝对值** 两档分级：`≥1e8` → `X.XX亿元`；`<1e8` → `X.XX万元`（含 `<1e4` 也按万元展示，如 `9890元` → `0.99万元`，**不**降到「元」）。换算后 **保留 2 位小数，四舍五入**。**逐值判定**：每个数值独立选档，**禁止**为"表格整齐"统一档位（同列允许出现「1.18亿元」与「8178.99万元」混排） |
| **比率** | `fieldType=2`（使用率等） | 保留 **2 位小数** + `%`（四舍五入） |
| **人数/HC** | `fieldType=1` | **整数**（四舍五入到个位）+ 量词「人」等，**不**用万/亿 |

**金额展示示例**（原值均为元）：

| 接口原值（元） | ✅ 正确 | ❌ 错误 |
|----------------|--------|---------|
| `117561400` | `1.18亿元` | `11756.14万元`（统一档位） |
| `81789889` | `8178.99万元` | — |
| `12345.6` | `1.23万元` | — |
| `9890` | `0.99万元` | `9890元`（降到元档） |

**覆盖**：用户明确要求「精确到元」「不要换算」时，可按原值（元）展示并注明单位，不再强制万/亿档位。

**终态 JSON**（内部流转；对用户用表格/短文）：

```json
{
  "status": "success | partial_success | failed",
  "summary": "一句话结论",
  "result": { "items": [], "totals": {} },
  "warnings": [],
  "errors": [],
  "meta": {
    "queryKey": "a3fe268e90104b459919c9f0c95dfbf6",
    "bizId": 119,
    "type": 1,
    "time_range": "string",
    "data_source": "北斗",
    "sql": null,
    "trace": null
  }
}
```

`type=3` 降级时 `meta.queryKey=null`、`meta.type=3`。

- [ ] 表格每一行（维度组合）是否都能在 MCP 返回的行数据中找到？输出行数是否 ≤ MCP 去重维度组合数？(铁律 #9)

### ‼️ 输出禁项铁律

面向用户的回答（表格、summary、口径说明）**禁止**包含以下内容；它们仅写入 `meta`，不进入用户可见正文：

1. **实现细节泄漏**：以下三类内容只写入 `meta`，不进入用户可见正文：
   - **技术标识**：`bizId`、`queryKey`、`北斗`、`人力成本域`、`data-query`、`trace_id`、模型名、服务名
   - **字段名/计算细节**：`mon`、`dept_id_lvl*`、`real_emp_cls_cd`、`ai_qa_*` 等 alias、SQL、计算公式、口径推导
   - **文档元注释**：本文档括号内"…未限定时不加 X 筛选…""…等价于全集…"等给模型看的判定逻辑
2. **未在下方「输出格式」模板中出现的栏目**（如「数据来源：xxx」「数据更新时间：xxx」）——模板没列就不写，禁止借用其他 skill 的输出习惯
3. **缺数月份的原因猜测**：成本为月度数据，**无 T+1 概念**；某月 `total=0` 时只写「暂无数据」，**禁止**附加「T+1 / 未更新 / 暂未发布 / 数据延迟」等任何原因解释
4. **跨部门数据替代**：用户查询的部门取数返回空（`total=0` / `noAuthMeasures` 全集 / 数据权限空）时，**禁止**改查上级/同级/下级部门数据后以「暂无数据」标注原部门来呈现；必须直接答复「{部门名}暂无成本数据」并停止，不展示任何替代部门的数值/表格。**condition 部门字段返回 403** 时同样禁止自动切换上级——这是「权限拒绝」非「数据为空」，必须告知具体权限错误并询问用户是否换上级部门，**不得擅自改查**
5. **金额档位违规**：金额必须**逐值**按 `≥1e8 / <1e8` 两档独立判定单位。**禁止**：① 把 `≥1e8` 的值写成「XXXX万元」（如 `1.18亿元` 写成 `11756.14万元`）；② 把 `<1e4` 的值降到「元」展示（如 `9890元`、`999.99元`），必须统一升到万元（`0.99万元`、`0.10万元`）；③ 为同列整齐统一档位（同列允许「1.18亿元」与「8178.99万元」混排）；④ 凭直觉换算不对照档位规则。输出每个金额前必须做一次 `|v| >= 1e8 ? 亿元 : 万元` 判定
6. **多指标单表展示**：所有指标列必须放在**同一张表**中，**禁止**因"列数过多""阅读宽度大"主动拆分为多张子表（如把"总览组"与"现金/股票明细组"拆为两张表）。多指标表的可读性问题由前端横向滚动解决，不在内容层处理。**列序约定**：维度列（月份/部门）→ 预算类金额（月度预算/现金预算/股票预算）→ 实际类金额（实际成本/现金实际/股票实际）→ 使用率类（使用率/现金使用率/股票使用率）→ 结余类，同类指标聚集；**人数 / HC 列默认不展示**，仅当用户明确问询时才加入，位置紧跟维度列之后。**例外**：预测查询（预测成本/预测年）按场景级双表展示（实际段表 + 预测段表 + 全年累计行），详见「双表场景」节
7. **指标列严格对齐用户问询**：表格列数 = 用户实际问的指标数（含同义指标，如"使用率/使用情况"对应 `ai_qa_cost_rate`）。**禁止**主动追加用户未问的指标列（最常见误加：在职人数列 `ai_qa_emp_cnt2`）。**例外**：用户问"使用率"等比率类指标时，可同时取对应分子/分母（如 `cost_amt`/`bgt_amt`）用于多月累计计算，但**仅用于计算，不展示为列**；用户问 HC 模糊统称（HC 使用率/使用详情/使用情况/整体情况）时，默认必含 4 指标（HC 使用率/月末在职/滚测 HC/结余 HC）+ 可自由追加其他相关 HC 指标，不视为主动追加（详见输出场景表）
8. ⛔ **第二次累计查询 `dimensions` 不得为空 + `mon` 上限 ≤ `max_cost_mon` 且与表 1 一致**：① 去 `mon` 后若 `dimensions` 变空（典型：单部门多月查询），必须把 condition 中的部门字段（`dept_id_lvl{N}` 优先）补入 `dimensions`。**示例**：condition 含 `dept_id_lvl0='MI'` 且原 `dimensions=["mon"]` → 第二次 `dimensions=["dept_id_lvl0"]` 而非 `[]`；违反将导致后端报错。② 主查询及二次累计查询的 `mon` 上限**必须 ≤ Step 0 返回的 `max_cost_mon`**（用户显式给月份的部分除外），且与表 1 实际展示月份一致——否则累计=含缺数月预算 vs 不含缺数月实际 → 使用率/结余失真
9. ⛔ **行级数据必须 1:1 来自 MCP 返回**（防维度值幻觉）：
   - 每行（维度组合）必须能在 MCP 返回中找到；输出前校验「行数 ≤ MCP 去重维度组合数」，不满足必须删回。反例：MCP 返回职级 13-19 共 7 行 → 禁止凭"常识/推测"补 20/21 级。
   - MCP **不含**的维度值直接不展示，不写"暂无数据"；"暂无数据"仅用于 MCP **含**该行但金额=0/null（与 #3 一致）。

```
📊 {指标名称}
时间范围：{实际有数据的最早月} ~ {实际有数据的最晚月}

| {维度} | {指标} |
|--------|--------|
| 值A    | 数值（金额按万/亿档位，见 Output Contract） |

**金额单位**：表格与 `summary` 中的金额须按 Output Contract 换算为 **万元 / 亿元**（两档；2 位小数、四舍五入），勿对大额直接展示「113,912,966 元」类长串，也勿对小额降到「元」展示；使用率用 `%`，人数用整数。

口径说明（按下列分支输出，**仅输出引号内文字**）：
- 未指定员工类型 + 金额/比率/使用率类指标 → "员工类型范围：正式员工 + 外包 + 实习生 + 其他"
- 未指定员工类型 + **HC 类指标**（人数，`fieldType=1`，含 `hc`/`_hc_` 字样如 `ai_qa_est_hc`、`ai_qa_bgt_hc`） → "员工类型范围：正式员工 + 外包A + 外包B"
- 用户指定员工类型 → "员工类型范围：{实际纳入的类型}"
```

**多月累计规则**：表格行数 ≥ 2（多月数据）时，末行追加「合计」行。**合计值必须由后端按"去 mon 维度"重新查询取得，禁止客户端对各月行求和、求平均、或自算分子/分母**。
- **取数方式**：在原查询基础上发**第二次** `data_query`，dimensions 去掉 `mon`、保留原有业务维度（如子部门下钻字段）；**如去 mon 后 dimensions 为空（单部门查询场景），必须把 condition 中筛选的部门字段（如 `dept_id_lvl1`）补入 dimensions——dimensions 不能为空，否则后端报错**；其余参数（measures / condition）不变；返回的单行（或子部门展开多行）即为各 measure 在区间内的累计值
  - ❌ 错误：`dimensions: []`（空数组，后端报错）
  - ✅ 正确：condition 含 `dept_id_lvl0='MI'` → 第二次 `dimensions: ["dept_id_lvl0"]`
- **金额类**（`fieldType=8`）：合计列取第二次查询返回值，与各月同单位、同精度（万/亿元）
- **使用率 / 比率类**（`fieldType=2`）：合计列取第二次查询返回值（后端按统一口径计算，权威），保留 2 位小数 + %；**禁止**对各月比率求和、求平均，也禁止客户端自算「累计分子/累计分母」
- **HC / 人数类**（`fieldType=1`）：合计列**留空**（人数累计语义不明，避免误导）。注意：HC/人数列**默认不展示**（详见铁律第 7 条），仅在用户明确问询时才出现，此时再适用合计留空规则
- **差异处理**：第二次查询的累计值与各月行客户端求和不一致时，**以第二次查询为准**，必要时在 `warnings` 标注差异（一般为后端口径差异，非异常）
- 单月查询不发第二次调用，不加合计行

## 双表场景：实际 + 预测（预测查询专用）

**触发**：用户原文含预测语义指标（「预算成本」「预算使用率」「预测成本」「预测年成本」「预测使用率」「预计结余」等），无论是否带年份。

**切分点 N = Step 0 返回的 `max_cost_mon` + 1**（如 Step 0 返回 `max_cost_mon=5` → N=6，实际段=1~5 月，预测段=6~12 月）。本场景**必发 Step 0** 且附带读 `y_fcst_desc` 用于预测段说明文案；Step 0 失败时降级用日历当前月推断 N。各表无内置合计行。

| 段 | 月份 | measures | 列 |
|----|------|---------|-----|
| 表 1 实际 | 1 ~ N-1 | `ai_qa_cost_amt` / `ai_qa_bgt_amt` / `ai_qa_cost_rate` / `ai_qa_rest_amt` | 月份 / 实际成本 / 预算 / 使用率 / 结余 |
| 表 2 预测 | N ~ 12 | `ai_qa_fcst_cost_amt` / `ai_qa_bgt_amt` / `ai_qa_fcst_cost_rate` / `ai_qa_fcst_rest_amt` | 月份 / 预测成本 / 预算 / 预测使用率 / 预测结余 |

**全年累计（1-12 月）— 仅 2 行**：
- 全年总成本 = 累计实际(1~N-1) + 累计预测(N~12)，两次「去 mon」客户端拼接（**仅此一处允许客户端聚合**）
- 全年总预算 = `ai_qa_bgt_amt` 后端累计

### 取数路径（最多 4 次 `data_query`，「去 mon」遵循铁律 #8）

| ③ | `ai_qa_fcst_cost_amt` | N ~ 12 |
| ④ | `ai_qa_bgt_amt` | 1-12 |

| ✅ 预测成本 / 预测年（含「预测成本/预测年成本/预测使用率/预计结余」等预测语义） | **双表 + 全年累计行**：表 1 实际段(1~N-1月，无合计行) + 表 2 预测段(N~12月，无合计行) + 全年累计行(详见下文「双表场景」节)。**N = Step 0 返回的 `max_cost_mon` + 1**（如 Step 0 返回 `max_cost_mon=5` → N=6，实际段=1~5 月，预测段=6~12 月） |
| ❌ 查询部门整体无数据 | 直接答「{部门名}暂无成本数据」，**禁止**用上级/同级/下级部门数据替代展示 |
| ✅ 子部门末月无数据 | 多月 + 子部门展开时，剔除查询区间末月 `total=0` 的子部门（视为已撤销），`warnings` 列出被剔除项 |
| ⚠️ 降级 | 说明使用了替代数据和原因（如部分指标无权限） |
| ❌ 失败 | 明确失败原因，给出解决方向 |
| ❓ 需确认 | 清晰列出缺失信息，一次最多问一个问题 |

## 异常与熔断

| 场景 | 处理 |
|------|------|
| MCP 连续 2 次失败 / 总耗时 >120s | 熔断，输出诊断包 |
| 单次调用 >30s | 中断并重试或失败 |
| 某月 `total=0` | 该月填「暂无数据」，标题/时间范围收敛至有数据的月份区间，**禁止**写 T+1/未更新等原因 |
| measures 含 `ai_qa_cost_amt` 且某月该字段**所有维度组合**均为 0/null/缺失（**Step 0 失败时的兜底**——主防线已是 Step 0 通过 `max_cost_mon` 在查询构造阶段裁剪 mon 上限） | 视该月成本数据未发生：① 表 1 该月行不展示，标题/时间范围收敛至有数据的月份区间；② **二次累计查询的 `mon` 条件必须同步剔除该月**（铁律 #8），否则累计口径=含缺数月预算 vs 不含缺数月实际 → 使用率失真；③ 双表场景的 `N`（实际/预测切分点）= 缺数月最早一个，而非日历当前月 |
| 查询部门 `total=0` / 数据权限空 / `noAuthMeasures` 全集 | 输出「{部门名}暂无成本数据」并停止，**禁止**改查上级/同级/下级部门 |
| Step 2 取数阶段 condition 部门字段返回 **403** | 输出「您当前没有「{部门名}」的成本数据查询权限，是否需要换用上级部门「{上级部门名}」查询？」并停止；**禁止**自动改查上级，必须用户显式同意后再查。**注**：Step 1 部门探查阶段的 403 不适用本规则——视为「未找到」继续探查下一层级 |
| 多月子部门展开，某子部门查询区间末月 `total=0` | 渲染时剔除该子部门，`warnings` 标注「已剔除末月无数据子部门：{...}」（视为已撤销/合并） |
| `noAuthMeasures` 非空 | `warnings` 列出 |
| 指标未知 | `data_search_measures` → 追问 |
| 意图模糊 | A/B/C 选项 |

**失败诊断包**（熔断时面向用户输出）：

```
🔴 查询失败
症状：{表现}
修复建议：{用户可操作项}
```

> 技术细节（bizId/queryKey/失败步骤号/已尝试 type=3 等）写入 `meta.warnings`，不进入用户可见正文（铁律 #1）。
