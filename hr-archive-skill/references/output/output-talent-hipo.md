# STAR_TALENT_HIPO — 高潜人才名单输出规范

## 角色定位

组织档案AI问答助手，针对高潜人才查询问题，基于人才标签体系的结构化查询结果，合并三个梯队在**一张飞书卡片**中输出人才名单。

## 输出结构

> **⚠️ 高潜报告以 1 张飞书卡片发送，三个梯队按顺序排列在同一张卡片中（核心高潜 → 高潜 → 潜力关注）。**
>
> 卡片顶部为总结说明段，之后按梯队依次输出 3 列表格（姓名 / 基本信息 / 高潜标识），梯队间用三级标题分隔。
> 高潜标识列展示：最近两次绩效、年龄分位（来自 tag_dscr）、最近一次晋升时间。
> 若某梯队无人，跳过该梯队段落。

---

## 一、飞书卡片发送流程

> **⚠️ 高潜名单必须使用 `msg_type=interactive` 发送飞书交互卡片，严禁使用 `msg_type=text` 或其他类型。**

### 发送步骤

1. **组装** Markdown 内容（总结段 + 各梯队标题与表格）
2. **组装**飞书卡片 JSON，使用 `exec` 工具调用 `send_card()` 函数发送（参考 [references/report/send-card.md](../report/send-card.md)）
3. `receive_id` **必须使用 runtime context 中的 `Chat ID`**（不是 `sender_open_id`）

### 飞书卡片 JSON 结构

```json
{
  "schema": "2.0",
  "header": {
    "title": {"tag": "plain_text", "content": "CARD_TITLE"},
    "template": "blue"
  },
  "body": {
    "elements": [
      {"tag": "markdown", "content": "CARD_CONTENT"}
    ]
  }
}
```

> **🚨🚨🚨 `CARD_CONTENT` 必须是填充后的完整 Markdown 文本，不能为空！**
>
> - `CARD_CONTENT` 是总结段 + 各梯队表格拼接后的完整 Markdown 字符串
> - **卡片内容为空 = 严重错误**：如果 `body.elements[0].content` 为空字符串 `""`、为 `"CARD_CONTENT"` 占位符原文、或未赋值，飞书卡片会显示为空白
> - **卡片显示 JSON 原文 = 严重错误**：如果把卡片 JSON 结构或查询结果数组放进了 `content` 字段，用户会看到 JSON 乱码而非表格
> - 正确做法：组装 Markdown 字符串 → 将**完整 Markdown 字符串**放入 `body.elements[0].content`
>
> 详细调用示例见 [references/report/send-card.md](../report/send-card.md)

### 完整调用示例

```python
token = get_feishu_token()

card = {
    "schema": "2.0",
    "header": {
        "title": {"tag": "plain_text", "content": "⭐ 中国区 · 高潜人才名单"},
        "template": "blue"
    },
    "body": {
        "elements": [
            {"tag": "markdown", "content": "共识别出 **30** 名高潜员工（⭐核心高潜 10 人，🌟高潜 12 人，📈潜力关注 8 人）。\n\n> 数据来源：人才标签体系 · 高潜标签维度\n> 数据截止：2026-05-18\n\n---\n\n### ⭐ 核心高潜（10 人）\n\n| 姓名 | 基本信息 | 高潜标识 |\n|------|---------|----------|\n| [张三（zhangsan1）](https://archive.hr.mioffice.cn/talent-details?entrySource=lark-card&entrance=3&userId=zhangsan1) | 15级 · 32.5岁 · 产品经理 · 手机部 | 最近两次绩效(半年/年度)：B+(2025)、B(2024)<br>年龄分位：研发序列15级年龄Top10%<br>最近一次晋升时间：2024年06月 |\n\n---\n\n### 🌟 高潜（12 人）\n\n| 姓名 | 基本信息 | 高潜标识 |\n|------|---------|----------|\n| ... |\n\n---\n\n### 📈 潜力关注（8 人）\n\n| 姓名 | 基本信息 | 高潜标识 |\n|------|---------|----------|\n| ... |"}
        ]
    }
}

result = send_card(card, "oc_xxx", token)
print(json.dumps(result, ensure_ascii=False))
```

> **⚠️ 常见报错：`content is not a string in json format`**
> 飞书 API 要求请求体中的 `content` 字段是 **JSON 字符串**（双重序列化）。
> `send_card` 函数中 `json.dumps(card)` 负责第一次序列化（卡片 dict → 字符串），外层 `json.dumps(...)` 负责第二次序列化（整个请求体 → 字符串）。
> **必须严格按照 `send-card.md` 中的写法，不要自行修改 payload 的构造方式。**

**统一规则：**
- `header.template` **固定为 `"blue"`**
- `CARD_CONTENT` 中换行必须使用 `\n`（JSON 转义换行符），严禁将多行内容拼接成一行——换行丢失会导致飞书无法渲染 Markdown 表格
- 卡片标题不要在内容中重复——`header.title` 已是卡片标题，`CARD_CONTENT` 中不要再以 `# 标题` 或加粗文字重复
- **禁止发送测试卡片**——发送失败时直接重试原始卡片，不得向用户会话发送测试内容

**内容 Markdown 结构**：

## 三、各列填写规则（3 列）

### 「姓名」列

- 格式：`[{emp_nm}（{oprid}）](https://archive.hr.mioffice.cn/talent-details?entrySource=lark-card&entrance=3&userId={oprid})`
- 链接文本为「姓名（账号）」，链接指向人才档案详情页
- `userId` 使用 SQL 返回的 `oprid`

### 「基本信息」列

- 格式：`{pos_lvl}级 · {age_y}岁 · {job_nm} · {dept_nm}`
- 四项信息用 ` · ` 分隔，顺序固定：职级 → 年龄 → 岗位名称 → 部门
- `age_y` 已由 SQL ROUND 到 1 位小数，直接使用
- `job_nm` 为岗位名称，直接使用 SQL 返回值
- 若某字段为空值/NULL，跳过该项（如无岗位名称：`15级 · 32.5岁 · 手机部`）

### 「高潜标识」列

- 三段信息**逐行展示**（用 `<br>` 换行），顺序固定：绩效 → 年龄分位 → 晋升时间
- **第 1 行 — 最近两次绩效**：格式 `最近两次绩效(半年/年度)：{late_pfm}({late_pfm_prd})、{late_late_pfm}({late_late_pfm_prd})`
  - 示例：`最近两次绩效(半年/年度)：B+(2025)、B(2024)`
  - 若 `late_pfm` 为空，显示 `最近两次绩效(半年/年度)：--`
  - 若 `late_late_pfm` 为空，仅显示最近一次：`最近两次绩效(半年/年度)：B+(2025)`
- **第 2 行 — 年龄分位**：从 `tag_dscr` 中提取 `序列X职级x年龄TopN%` 部分
  - `tag_dscr` 原始格式为：`最近两次绩效：A级及以上 + 序列X职级x年龄Top10%`
  - 提取规则：取 `+` 号后面的内容（去除前后空格），即 `序列X职级x年龄TopN%` 部分
  - 展示格式：`年龄分位：序列X职级x年龄Top10%`
  - 若 `tag_dscr` 为空或无法提取分位信息，跳过此行
- **第 3 行 — 最近一次晋升时间**：将 `late1_pro_dt` 格式化为 `最近一次晋升时间：YYYY年MM月`
  - 示例：`最近一次晋升时间：2024年06月`
  - 若 `late1_pro_dt` 为空，跳过此行
- **空值处理**：仅绩效为必填（至少显示第 1 行），其他两行为空时跳过
- 完整示例（单元格内 3 行）：
  ```
  最近两次绩效(半年/年度)：B+(2025)、B(2024)
  年龄分位：研发序列15级年龄Top10%
  最近一次晋升时间：2024年06月
  ```
- 仅有绩效的示例（单元格内 1 行）：
  ```
  最近两次绩效(半年/年度)：B+(2025)、B(2024)
  ```

## 四、输出铁律（最高优先级）

### 完整性铁律

1. **所有有人的梯队必须输出**——该梯队有 N 人，就必须输出 N 行表格，禁止截断、省略或用"..."代替（核心高潜超 200 人的截断除外）
2. **所有梯队使用完全相同的 3 列结构**——禁止对任何梯队简化列数或减少内容
3. **梯队内排序**：按 pos_lvl DESC, age_y ASC（职级高→年龄小优先，与 SQL ORDER BY 一致）
4. **空梯队不输出**——如果某梯队 0 人则跳过该段落，不输出空表格
5. **核心高潜超 200 人截断**：仅保留前 200 条，总结段注明"核心高潜共 N 人，当前展示前 200 人"

### 格式铁律

6. **表格前后必须各有一个空行**——否则飞书无法解析 Markdown 表格
7. **行内格式标记必须成对闭合**——未闭合的 `**` 会导致后续内容渲染异常
8. **高潜标识列使用 `<br>` 换行**——该列内绩效、年龄分位、晋升时间三行信息用 `<br>` 分隔；姓名列和基本信息列禁止换行
9. **禁止添加序号列**——表格固定 3 列（姓名/基本信息/高潜标识），不加额外列
10. **各梯队段落之间用 `---` 分隔线隔开**
11. **三级标题格式固定**：`### {emoji} {梯队名}（{人数} 人）`
12. **表格前后不要出现引用符号**（`>`）——会影响飞书卡片 Markdown 渲染，导致表格被识别为 blockquote 而非 table（总结段的引用除外）

### 数据铁律

13. **数据驱动**：所有内容基于 `sql_query` 工具返回的真实数据，禁止编造
14. **禁止暴露技术细节**：表名、字段名、SQL 等技术信息不得出现在输出中
15. **薪酬相关信息禁止展示**

### 绝对禁止

- ❌ 将表格转为纯文字描述
- ❌ 省略任何有数据的梯队
- ❌ 截断表格行数（核心高潜 200 截断规则除外）
- ❌ 省略表格中的任何列
- ❌ 自行编写 SQL 查询数据（必须通过 `sql_query` 工具）
- ❌ 输出中间推理过程
- ❌ 在文字消息中重复输出卡片内容
- ❌ 使用 `msg_type=text` 发送（必须 `msg_type=interactive`）
- ❌ 发送测试卡片进行调试或验证
- ❌ 因"内容过长"而主动缩减表格行数
- ❌ 将三个梯队拆分为多张卡片

### 格式自检清单（发送前必检）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 表格列数 | 每行恰好 3 列（含两侧 `|`共 4 个） |
| 2 | `**` 闭合 | 每行 `**` 出现偶数次 |
| 3 | 单元格换行规范 | 高潜标识列用 `<br>` 换行；姓名列和基本信息列禁止换行 |
| 4 | 表格前空行 | 表格 `|` 行上方有空行 |
| 5 | 表格后空行 | 表格最后一行下方有空行 |
| 6 | 人数一致 | 各梯队表格行数 = 总结段声明的对应人数 |
| 7 | 高潜标识格式 | 每行高潜标识列包含「最近两次绩效(半年/年度)：」开头的绩效信息，各项用 `<br>` 换行 |
| 8 | 姓名超链接 | 每行姓名列为 `[姓名（账号）](https://archive.hr.mioffice.cn/...)` 格式 |
| 9 | 卡片类型 | `msg_type` 为 `interactive`，非 `text` |
| 10 | 卡片内容非空 | `body.elements[0].content` 为非空 Markdown 字符串 |
| 11 | 换行符保留 | 内容中 `\n` 分隔各段落和表格行，未被拼接成一行 |
| 12 | 颜色统一 | `header.template` 为 `"blue"` |

**任何一条未通过，禁止发送，必须修正后再发送。**

## 五、无数据 / 权限不足

不发送卡片，直接输出纯文本：

```
抱歉，您的权限暂时无法查询到相关信息~
可能原因：1.查询内容涉及本人信息，目前不支持个人自查；2.查询范围超出了您当前的数据权限范围。
```

## 六、发送规则

> **⚠️ 整个高潜报告仅发送 1 张卡片，三个梯队合并在同一张卡片内。禁止拆分为多张卡片。**

### 发送完毕

卡片发送后，回复："已为您生成{部门名称}高潜人才名单。"
**禁止**在文字消息中重复输出报告内容或 JSON。
**禁止发送测试卡片**——发送失败时直接重试原始卡片，或向用户报告发送失败。
